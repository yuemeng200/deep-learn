# 第4章：视频帧处理——让画面动起来

前三章我们用 Wasm 处理的是静态图像——加载一张图片，应用滤镜，看到结果。但视频编辑器的核心是**视频**，是连续流动的画面。本章我们将跨越这道门槛：从 Canvas 中逐帧提取视频数据，送入 Wasm 管线处理，再实时渲染回画面。核心本质"JS 管调度，Wasm 管计算"将在这里展现得淋漓尽致——JS 用 `requestAnimationFrame` 驱动帧循环，Wasm 在每一帧中完成像素级处理。

---

## 4.1 视频就是一堆图片

视频的本质极其简单：**一串连续的图片，以固定速率播放。**

```
视频 = [帧0] [帧1] [帧2] ... [帧N]
        ↓
       每秒 30 帧（30fps）
       每帧就是一张 RGBA 图片
```

这意味着：
- **处理视频 = 处理每一帧图片**
- 我们在前两章构建的图像管线，可以直接用于视频帧
- 新增的挑战不是"怎么处理像素"，而是"怎么高效地把视频帧送过来"

### 浏览器中的视频处理流水线

```
<video> 元素
    ↓ 播放视频流
Canvas 2D 上下文
    ↓ drawImage(video, ...)    ← 把当前帧画到 Canvas
    ↓ getImageData()           ← 提取像素数据
Wasm 管线
    ↓ process()                ← 像素处理
Canvas 2D 上下文
    ↓ putImageData()           ← 写回处理后的帧
    ↓ 显示给用户
```

关键认知：**浏览器不会自动把视频帧给你**。你需要主动在每一帧"抢"数据——这就是 `requestAnimationFrame` 的用武之地。

---

## 4.2 requestAnimationFrame：浏览器的帧循环引擎

`requestAnimationFrame`（简称 rAF）是浏览器提供的帧调度 API：

```javascript
function frameLoop() {
    // 1. 从 <video> 抓取当前帧
    ctx.drawImage(video, 0, 0);

    // 2. 提取像素，送入 Wasm 处理
    const imageData = ctx.getImageData(0, 0, width, height);
    pipeline.set_filter(brightness, contrast, saturation, grayscale);
    const result = pipeline.process();

    // 3. 写回 Canvas 显示
    imageData.data.set(result);
    ctx.putImageData(imageData, 0, 0);

    // 4. 请求下一帧
    requestAnimationFrame(frameLoop);
}

// 启动循环
requestAnimationFrame(frameLoop);
```

浏览器会在每次屏幕刷新前调用这个回调（通常 60 次/秒）。但如果 Wasm 处理一帧耗时超过 16.6ms，帧率就会下降——这正是下一章性能优化的动力。

## 4.3 实战：实时视频特效预览器

### 步骤一：扩展 Rust 管线，支持零拷贝输出

上一章结尾的思考题已经提示：`process()` 返回 `Vec<u8>` 会有复制开销。对实时视频来说，每帧 8MB 的复制不可接受。优化管线：

```rust
use wasm_bindgen::prelude::*;

#[wasm_bindgen]
extern "C" {
    #[wasm_bindgen(js_namespace = console)]
    fn log(s: &str);
}

macro_rules! console_log {
    ($($t:tt)*) => (log(&format_args!($($t)*).to_string()))
}

struct FilterConfig {
    brightness: i16,
    contrast: f32,
    saturation: f32,
    grayscale: bool,
}

#[wasm_bindgen]
pub struct Pipeline {
    width: u32,
    height: u32,
    output: Vec<u8>,
    config: FilterConfig,
    frame_count: u32,
}

#[wasm_bindgen]
impl Pipeline {
    #[wasm_bindgen(constructor)]
    pub fn new(width: u32, height: u32) -> Self {
        let size = (width * height * 4) as usize;
        Pipeline {
            width,
            height,
            output: vec![0u8; size],
            config: FilterConfig {
                brightness: 0,
                contrast: 1.0,
                saturation: 1.0,
                grayscale: false,
            },
            frame_count: 0,
        }
    }

    /// 设置滤镜参数
    pub fn set_filter(
        &mut self,
        brightness: i16,
        contrast: f32,
        saturation: f32,
        grayscale: bool,
    ) {
        self.config = FilterConfig {
            brightness,
            contrast,
            saturation,
            grayscale,
        };
    }

    /// 处理一帧：输入数据原地处理
    pub fn process_frame(&mut self, data: &mut [u8]) -> u32 {
        self.frame_count += 1;

        let c = &self.config;
        for chunk in data.chunks_exact_mut(4) {
            let mut r = chunk[0] as f32;
            let mut g = chunk[1] as f32;
            let mut b = chunk[2] as f32;

            if c.grayscale {
                let gray = 0.299 * r + 0.587 * g + 0.114 * b;
                r = gray;
                g = gray;
                b = gray;
            }

            if c.brightness != 0 {
                let br = c.brightness as f32;
                r = (r + br).clamp(0.0, 255.0);
                g = (g + br).clamp(0.0, 255.0);
                b = (b + br).clamp(0.0, 255.0);
            }

            if c.contrast != 1.0 {
                r = ((r - 128.0) * c.contrast + 128.0).clamp(0.0, 255.0);
                g = ((g - 128.0) * c.contrast + 128.0).clamp(0.0, 255.0);
                b = ((b - 128.0) * c.contrast + 128.0).clamp(0.0, 255.0);
            }

            if c.saturation != 1.0 {
                let gray = 0.299 * r + 0.587 * g + 0.114 * b;
                r = (gray + (r - gray) * c.saturation).clamp(0.0, 255.0);
                g = (gray + (g - gray) * c.saturation).clamp(0.0, 255.0);
                b = (gray + (b - gray) * c.saturation).clamp(0.0, 255.0);
            }

            chunk[0] = r as u8;
            chunk[1] = g as u8;
            chunk[2] = b as u8;
        }

        self.frame_count
    }

    /// 获取已处理的帧数
    pub fn frame_count(&self) -> u32 {
        self.frame_count
    }

    /// 重置帧计数
    pub fn reset_counter(&mut self) {
        self.frame_count = 0;
    }
}
```

关键改变：`process_frame` 接收 `&mut [u8]`，**直接在原始数据上原地修改**，不返回任何东西。JS 侧的 `Uint8ClampedArray` 和 Wasm 侧的 `&mut [u8]` 指向同一块内存——**全程零拷贝**。

## 4.4 帧处理的关键细节

### 4.4.1 `willReadFrequently` 提示

注意 Canvas 创建时的参数：

```javascript
const ctx = canvas.getContext('2d', { willReadFrequently: true });
```

这是一个性能提示。默认情况下，浏览器的 Canvas 使用 GPU 加速渲染，但 `getImageData()` 需要从 GPU 读回 CPU，这很慢。设置 `willReadFrequently: true` 告诉浏览器"我会频繁读取像素数据"，浏览器会优化为 CPU 侧的软件渲染，避免 GPU-CPU 的来回传输。

### 4.4.2 零拷贝的关键路径

```
ctx.drawImage(video)    → 像素写入 Canvas 内部缓冲区
ctx.getImageData()      → 返回 ImageData（浏览器内部管理）
imageData.data          → Uint8ClampedArray（JS 侧）
pipeline.process_frame(imageData.data)
    → wasm-bindgen 把 Uint8ClampedArray 映射为 &mut [u8]
    → Rust 原地修改
    → 返回 JS
ctx.putImageData()      → 写回 Canvas 显示
```

全程只有 `getImageData` 和 `putImageData` 涉及数据搬运（Canvas 内部 ↔ JS），Wasm 处理环节完全零拷贝。

### 4.4.3 帧率与处理时间的关系

```
目标帧率    每帧预算
30 fps     33.3 ms
60 fps     16.6 ms

每帧预算 = 1000ms / 目标帧率

实际每帧耗时 = drawImage + getImageData + wasm_process + putImageData
```

如果 Wasm 处理一帧 1080p 需要约 20ms，加上 Canvas 操作约 5ms，总计 25ms——能跑 30fps 但到不了 60fps。下一章我们会把处理时间压到 5ms 以内。

## 4.5 从"预览"到"编辑"：帧的精细化控制

当前的预览器只是把视频逐帧播放并加滤镜。一个真正的编辑器需要更多控制：

| 能力 | 当前状态 | 编辑器需要 |
|------|---------|-----------|
| 播放/暂停 | ✓ | ✓ |
| 帧级别定位 | ✗ | 可以跳到任意帧 |
| 多滤镜组合 | ✓ | ✓ |
| 多片段拼接 | ✗ | 时间轴上排列多个片段 |
| 转场效果 | ✗ | 片段之间有淡入淡出等过渡 |

帧级定位是实现时间轴的基础。利用 `<video>` 元素的 `currentTime` 属性可以实现逐帧跳转：

```javascript
// 跳到第 N 帧
function seekToFrame(frameNumber) {
    const frameDuration = 1 / 30; // 假设 30fps
    video.currentTime = frameNumber * frameDuration;
}

video.addEventListener('seeked', () => {
    // seek 完成后渲染当前帧
    ctx.drawImage(video, 0, 0, canvas.width, canvas.height);
    // ... 应用滤镜 ...
});
```

多片段拼接和转场效果会在第7章的项目整合中实现。

## 章末思考题

**题目一**：当前的视频处理流程中，`getImageData()` 和 `putImageData()` 是不可避免的 Canvas API 调用。它们各自大约需要多少时间？如果视频是 4K（3840×2160），这两个操作的耗时会如何变化？

**参考答案**：

对于 1080p 视频（约 8MB 像素数据），`getImageData()` 和 `putImageData()` 各约 1-3ms（使用 `willReadFrequently` 模式）。4K 视频数据量是 1080p 的 4 倍（约 33MB），耗时也会接近 4 倍，即各 4-12ms。

两步合计 8-24ms，仅 Canvas 操作就可能吃掉 30fps 帧预算的一半以上。这就是为什么专业视频编辑器通常使用 **OffscreenCanvas + Web Worker**——在后台线程完成所有像素操作，主线程只负责显示最终结果。下一章会详细介绍。

---

**题目二**：`requestAnimationFrame` 的回调频率取决于屏幕刷新率（通常 60Hz），但视频本身可能是 24fps、30fps 或 60fps。如果视频是 24fps 而屏幕是 60Hz，会发生什么？会浪费计算吗？

**参考答案**：

会。`requestAnimationFrame` 以 60fps 调用回调，但 24fps 视频每秒只有 24 帧新数据。这意味着 60 次调用中，大约有 36 次处理的是**同一帧**——白白浪费了 Wasm 的计算能力。

优化方案：

```javascript
let lastVideoTime = -1;

function frameLoop() {
    // 检查视频是否前进到了新的一帧
    if (video.currentTime === lastVideoTime) {
        // 同一帧，跳过处理
        animFrameId = requestAnimationFrame(frameLoop);
        return;
    }
    lastVideoTime = video.currentTime;

    // 只有新帧才处理
    ctx.drawImage(video, 0, 0);
    // ...
}
```

但 `currentTime` 的精度可能不够（浮点数比较），更可靠的方式是基于时间计算当前帧号：

```javascript
const currentFrame = Math.floor(video.currentTime * fps);
if (currentFrame === lastFrame) { /* 跳过 */ }
```

---

**题目三**：目前我们用 `<video>` 元素加载视频文件。但 `<video>` 会自动解码视频帧。如果视频编码是 H.264，解码本身就是在做大量计算（运动估计、熵解码等）。浏览器的视频解码是在哪里完成的？和我们 Wasm 的像素处理会抢 CPU 吗？

**参考答案**：

浏览器的视频解码通常在**单独的解码线程**或**GPU 硬件解码器**上完成，不会和我们 Wasm 的主线程像素处理直接抢 CPU。这就是为什么即使 Wasm 在忙，视频播放本身不会卡顿——解码和渲染是流水线化的。

但在低端设备上，如果 GPU 不可用，解码会回退到软件解码（CPU），此时解码和 Wasm 处理会竞争 CPU 时间。这种情况下，可以在 Wasm 中降低处理分辨率（比如只处理 1/4 大小的像素），用低分辨率预览保流畅，导出时再全分辨率处理。

现代 WebCodecs API（第6章）给了我们更精细的控制——可以直接拿到解码后的原始帧数据，绕过 `<video>` 元素。
