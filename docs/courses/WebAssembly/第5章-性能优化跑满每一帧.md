# 第5章：性能优化——跑满每一帧

前四章我们从零搭建了一个视频帧处理管线，能在浏览器中实时对视频施加滤镜。但如果你用 1080p 视频试过，可能已经注意到帧率不太稳定——有时掉到 30fps 以下，滑块拖动时感觉有延迟。本章我们要直面性能瓶颈，用 SIMD 并行指令、Web Worker 离线程、内存池化三大武器，把处理速度提升数倍。核心本质"让浏览器拥有接近原生的计算能力"——优化就是把这个"接近"推到极致。

---

## 5.1 先测量，再优化

优化之前，必须知道时间花在哪里。上一章我们已经在 `frameLoop` 中加了计时，现在做一个更精确的分析：

```javascript
function frameLoop() {
    const t0 = performance.now();

    ctx.drawImage(video, 0, 0, canvas.width, canvas.height);
    const t1 = performance.now();

    const imageData = ctx.getImageData(0, 0, canvas.width, canvas.height);
    const t2 = performance.now();

    pipeline.process_frame(imageData.data);
    const t3 = performance.now();

    ctx.putImageData(imageData, 0, 0);
    const t4 = performance.now();

    console.log(`draw: ${(t1-t0).toFixed(1)}ms | ` +
                `getImage: ${(t2-t1).toFixed(1)}ms | ` +
                `wasm: ${(t3-t2).toFixed(1)}ms | ` +
                `put: ${(t4-t3).toFixed(1)}ms | ` +
                `total: ${(t4-t0).toFixed(1)}ms`);

    animFrameId = requestAnimationFrame(frameLoop);
}
```

在 1080p（1920×1080）视频上，典型数据：

```
draw: 0.3ms | getImage: 1.5ms | wasm: 18-25ms | put: 1.0ms | total: 21-28ms
```

**Wasm 处理占了 80%+ 的时间。** 这正是我们优化的目标。

## 5.2 武器一：SIMD——一次处理多个像素

### 5.2.1 SIMD 是什么

SIMD（Single Instruction, Multiple Data）——一条指令，同时处理多个数据。

```
普通操作（标量）：
  add pixel[0] → 结果[0]
  add pixel[1] → 结果[1]
  add pixel[2] → 结果[2]
  add pixel[3] → 结果[3]
  需要 4 条指令

SIMD 操作（向量）：
  add [pixel[0], pixel[1], pixel[2], pixel[3]] → [结果[0], 结果[1], 结果[2], 结果[3]]
  只需 1 条指令
```

Wasm SIMD 支持 128 位向量，可以一次处理 4 个 32 位值（正好是一个 RGBA 像素），或 16 个 8 位值。

### 5.2.2 Rust 中的 SIMD

Rust 通过 `std::arch` 内联函数或 `packed_simd` crate 提供对 SIMD 的访问。但在 Wasm 中，更实际的方式是让编译器**自动向量化**——写好标量代码，让 LLVM 自动利用 SIMD 指令。

```rust
// 关键：确保循环模式对编译器友好
fn process_chunk_simd_friendly(data: &mut [u8], brightness: i16) {
    // 确保 4 字节对齐（RGBA 对齐）
    for chunk in data.chunks_exact_mut(4) {
        for i in 0..3 {
            let val = chunk[i] as i16 + brightness;
            chunk[i] = val.clamp(0, 255) as u8;
        }
    }
}
```

LLVM 能自动识别这种模式并生成 SIMD 指令。但我们可以手动使用 Wasm SIMD intrinsics 获得更好的控制：

```rust
use wasm_bindgen::prelude::*;

// Wasm SIMD 通过 target_feature 启用
#[wasm_bindgen]
pub fn adjust_brightness_simd(data: &mut [u8], factor: i16) {
    // 确保 Wasm SIMD 可用
    #[cfg(target_feature = "simd128")]
    {
        use std::arch::wasm32::*;

        let brightness = i32x4_splat(factor as i32);
        let zero = i32x4_splat(0);
        let max_val = i32x4_splat(255);

        for chunk in data.chunks_exact_mut(16) {
            // 一次加载 16 字节（4 个像素的 R 或 G 或 B...）
            let pixels = v128_load(chunk.as_ptr() as *const v128);

            // 转为 i32x4 做运算
            let low = i32x4_extend_low_u16x8(u16x8_extend_low_u8x16(pixels));
            let high = i32x4_extend_high_u16x8(u16x8_extend_high_u8x16(pixels));

            // 加亮度
            let low_result = i32x4_add(low, brightness);
            let high_result = i32x4_add(high, brightness);

            // clamp 到 [0, 255]
            let low_clamped = i32x4_min(i32x4_max(low_result, zero), max_val);
            let high_clamped = i32x4_min(i32x4_max(high_result, zero), max_val);

            // 转回 u8 并存储
            // ...（实际代码需要更多 pack 操作）
        }
    }

    // fallback：无 SIMD 时的标量处理
    #[cfg(not(target_feature = "simd128"))]
    {
        for chunk in data.chunks_exact_mut(4) {
            for i in 0..3 {
                let val = chunk[i] as i16 + factor;
                chunk[i] = val.clamp(0, 255) as u8;
            }
        }
    }
}
```

> 实际项目中，手动写 SIMD intrinsics 又长又容易出错。更务实的方案是**依赖编译器自动向量化**，同时用 `wasm-opt` 在构建时做 SIMD 优化。下面介绍更实用的做法。

### 5.2.3 实用方案：编译器自动向量化 + wasm-opt

第一步：确保 Rust 编译时启用 SIMD：

```bash
# 在 Cargo.toml 中设置 profile
[profile.release]
opt-level = 3
lto = true

# 编译时显式启用 SIMD target feature
RUSTFLAGS='-C target-feature=+simd128' wasm-pack build --target web --release
```

第二步：用 `wasm-opt` 进一步优化：

```bash
# 安装 wasm-opt（属于 binaryen 工具包）
# macOS: brew install binaryen
wasm-opt -O3 -o pkg/video_editor_engine_bg.wasm pkg/video_editor_engine_bg.wasm
```

`wasm-opt` 会分析 Wasm 二进制，识别可以向量化的循环，自动生成 SIMD 指令。通常能把纯像素处理的速度提升 **2-4 倍**。

### 5.2.4 帮编译器帮你：写对 SIMD 友好的代码

编译器能否自动向量化，取决于你的代码模式。以下原则能让自动向量化更容易成功：

```rust
// ✓ 好的模式：简单循环，固定步长，无分支
for chunk in data.chunks_exact_mut(4) {
    let r = chunk[0] as f32;
    let g = chunk[1] as f32;
    let b = chunk[2] as f32;
    chunk[0] = (r * 0.299 + g * 0.587 + b * 0.114) as u8;
    chunk[1] = (r * 0.299 + g * 0.587 + b * 0.114) as u8;
    chunk[2] = (r * 0.299 + g * 0.587 + b * 0.114) as u8;
}

// ✗ 差的模式：有条件分支（编译器难以向量化）
for chunk in data.chunks_exact_mut(4) {
    if chunk[0] > 128 {  // 分支！
        chunk[0] = 255;
    } else {
        chunk[0] = chunk[0] * 2;
    }
}

// ✓ 把分支改为无分支计算
for chunk in data.chunks_exact_mut(4) {
    let val = chunk[0] as u16;
    let result = if val > 128 { 255 } else { val * 2 };
    // 编译器可以用 SIMD 的 min/max/select 指令替代分支
}
```

## 5.3 武器二：Web Worker——解放主线程

即使 Wasm 处理速度再快，只要在主线程执行，就会阻塞 UI。用户拖动滑块时感觉"卡"，不是因为处理慢，而是因为主线程被占满，没空响应交互。

### 5.3.1 架构变化

```
当前架构（单线程）：
主线程: [drawImage] [getImageData] [WASM处理] [putImageData] [等待下一帧]
                                    ↑ 这里阻塞了 UI 响应

优化架构（双线程）：
主线程:   [drawImage] → 发送帧数据 → ... → 接收结果 → [putImageData]
Worker 线程:              接收帧数据 → [WASM处理] → 发送结果
```

主线程和 Worker 通过 `postMessage` 通信。关键问题：传递 ImageData 需要复制数据吗？

### 5.3.2 Transferable Objects——零拷贝线程通信

好消息：`ArrayBuffer` 和 `ImageData` 是 **Transferable Objects**，可以从一个线程"转移"到另一个线程，而不需要复制：

```javascript
// 主线程 → Worker：转移 ArrayBuffer（零拷贝）
const buffer = imageData.data.buffer;
worker.postMessage({ type: 'process', buffer, width, height }, [buffer]);
// 注意第二个参数 [buffer] ——这是 transfer list
// 转移后，主线程的 imageData.data 变为空（detached）
```

```javascript
// Worker 线程
self.onmessage = (e) => {
    const { buffer, width, height } = e.data;
    const data = new Uint8Array(buffer);

    // Wasm 处理
    pipeline.set_filter(...);
    pipeline.process_frame(data);

    // 转移回去
    self.postMessage({ type: 'result', buffer }, [buffer]);
};
```

### 5.3.3 实战：创建 Wasm Worker

创建 `worker.js`：

```javascript
// worker.js — Web Worker 线程
import init, { Pipeline } from './pkg/video_editor_engine.js';

let pipeline = null;

self.onmessage = async (e) => {
    const { type } = e.data;

    if (type === 'init') {
        await init();
        const { width, height } = e.data;
        pipeline = new Pipeline(width, height);
        self.postMessage({ type: 'ready' });
        return;
    }

    if (type === 'process') {
        const { buffer, brightness, contrast, saturation, grayscale } = e.data;
        const data = new Uint8Array(buffer);

        pipeline.set_filter(brightness, contrast, saturation, grayscale);
        pipeline.process_frame(data);

        // 处理完毕，转移 buffer 回主线程
        self.postMessage({ type: 'result', buffer }, [buffer]);
        return;
    }

    if (type === 'reset') {
        if (pipeline) pipeline.reset_counter();
    }
};
```

更新 `index.html` 中的帧循环（关键改动）：

```javascript
// 主线程代码
const worker = new Worker('./worker.js', { type: 'module' });

let pendingBuffer = null;

worker.onmessage = (e) => {
    if (e.data.type === 'ready') {
        console.log('Worker 已就绪');
    }
    if (e.data.type === 'result') {
        // 收到处理完毕的帧
        pendingBuffer = e.data.buffer;
    }
};

// 初始化 Worker
worker.postMessage({ type: 'init', width: video.videoWidth, height: video.videoHeight });

function frameLoop() {
    if (!isPlaying) return;

    // 1. 从 video 抓帧
    ctx.drawImage(video, 0, 0, canvas.width, canvas.height);
    const imageData = ctx.getImageData(0, 0, canvas.width, canvas.height);

    // 2. 把帧数据转移给 Worker（零拷贝）
    const buffer = imageData.data.buffer;
    const brightness = parseInt(document.getElementById('brightness').value);
    const contrast = parseInt(document.getElementById('contrast').value) / 100;
    const saturation = parseInt(document.getElementById('saturation').value) / 100;

    worker.postMessage({
        type: 'process',
        buffer,
        brightness, contrast, saturation,
        grayscale: isGrayscale,
    }, [buffer]); // Transfer!

    // 3. 如果有已处理完的帧，显示它
    if (pendingBuffer) {
        const result = new ImageData(
            new Uint8ClampedArray(pendingBuffer),
            canvas.width,
            canvas.height
        );
        ctx.putImageData(result, 0, 0);
        pendingBuffer = null;
    }

    animFrameId = requestAnimationFrame(frameLoop);
}
```

### 5.3.4 双缓冲：消除等待

上面的实现有一个问题：主线程发送一帧给 Worker 后，要等到 Worker 处理完才能收到结果。如果 Worker 处理需要 20ms，主线程会有一帧显示旧数据。

解决方案是**双缓冲**——维护两个缓冲区，一个给 Worker 处理，一个给主线程显示：

```
缓冲区 A: Worker 正在处理（第 N+1 帧）
缓冲区 B: 主线程正在显示（第 N 帧的结果）
→ Worker 完成 → 交换 A/B → 主线程显示 B，Worker 开始处理下一帧到 A
```

```javascript
let buffers = [
    new ArrayBuffer(canvas.width * canvas.height * 4),
    new ArrayBuffer(canvas.width * canvas.height * 4),
];
let displayIndex = 0;
let processingIndex = 1;

worker.onmessage = (e) => {
    if (e.data.type === 'result') {
        // Worker 完成了 processingIndex 对应的缓冲区
        buffers[processingIndex] = e.data.buffer;
        // 交换
        [displayIndex, processingIndex] = [processingIndex, displayIndex];
    }
};
```

## 5.4 武器三：内存池化

每次 `getImageData()` 都会分配新的内存，`postMessage` 传输后旧内存被回收。这个分配-回收循环会增加 GC 压力。

### 内存池方案

预先分配固定数量的缓冲区，循环使用：

```javascript
class BufferPool {
    constructor(bufferSize, count = 3) {
        this.pool = Array.from({ length: count }, () => new ArrayBuffer(bufferSize));
        this.available = [...this.pool];
    }

    acquire() {
        if (this.available.length === 0) return null;
        return this.available.pop();
    }

    release(buffer) {
        this.available.push(buffer);
    }
}

// 使用
const pool = new BufferPool(canvas.width * canvas.height * 4, 3);
```

配合 Worker，流程变为：

```javascript
function frameLoop() {
    const buffer = pool.acquire();
    if (!buffer) {
        // 没有空闲缓冲区（Worker 还没处理完），跳过这帧
        animFrameId = requestAnimationFrame(frameLoop);
        return;
    }

    // 复制帧数据到池中缓冲区
    ctx.drawImage(video, 0, 0);
    const imageData = ctx.getImageData(0, 0, canvas.width, canvas.height);
    new Uint8Array(buffer).set(imageData.data);

    // 发给 Worker
    worker.postMessage({ type: 'process', buffer, ... }, [buffer]);
}
```

## 章末思考题

**题目一**：我们用 Transferable Objects 在主线程和 Worker 之间零拷贝传递帧数据。但 `postMessage` 本身有开销——序列化消息元数据（type、brightness 等字段）。在高频场景（每秒 60 次 postMessage），这个开销大吗？有没有更激进的优化方案？

**参考答案**：

`postMessage` 的开销来自两部分：
1. **结构化克隆**：消息对象本身的序列化。简单对象（几个数字+字符串）开销约 0.05-0.1ms，60fps 下可忽略
2. **Transfer**：ArrayBuffer 转移只是改所有权指针，几乎零开销

总计每帧约 0.1-0.2ms 的通信开销，相比 20ms 的处理时间微不足道。

更激进的方案是**SharedArrayBuffer**——主线程和 Worker 真正共享同一块内存，不需要 postMessage：

```javascript
// 主线程
const shared = new SharedArrayBuffer(width * height * 4);
const view = new Uint8Array(shared);
worker.postMessage({ type: 'init', shared });

// 写入数据后通知 Worker
Atomics.notify(new Int32Array(shared), 0); // 信号量通知
```

Worker 通过 `Atomics.wait()` 阻塞等待通知。这消除了所有通信开销，但需要处理同步问题（不能同时读写同一位置）。SharedArrayBuffer 还需要特殊的 HTTP 响应头（`Cross-Origin-Opener-Policy` 和 `Cross-Origin-Embedder-Policy`），部署稍复杂。

---

**题目二**：SIMD 向量化让像素处理提速了 2-3 倍。但灰度化公式中，三个通道的计算方式不同（权重不同），编译器还能向量化吗？如果能，它是怎么做到的？

**参考答案**：

能。虽然三个权重不同，但它们是**常量**。编译器把三个乘法视为：

```
result = R * 0.299 + G * 0.587 + B * 0.114
       = R * const_r + G * const_g + B * const_b
```

这是一个标准的**乘加（FMA）**运算，SIMD 指令原生支持。编译器只需要把三个常量加载到向量寄存器，然后对 4 个像素同时执行：

```
向量操作：[R0,R1,R2,R3] * [0.299, 0.299, 0.299, 0.299]
        + [G0,G1,G2,G3] * [0.587, 0.587, 0.587, 0.587]
        + [B0,B1,B2,B3] * [0.114, 0.114, 0.114, 0.114]
```

4 个像素的灰度值同时算出。关键在于：**"不同"不等于"不能并行"——只要操作的结构相同（都是乘加），只是系数不同，SIMD 就能处理。**

---

**题目三**：本章的优化都集中在"让单帧处理更快"。但从系统角度看，还有一个优化维度：减少需要处理的帧数。在什么场景下可以跳过某些帧不处理？这样做有什么副作用？

**参考答案**：

可跳帧的场景：

1. **视频静止区域**：前后帧差异极小的画面（如风景固定镜头、访谈节目），可以检测帧差，只处理变化的部分
2. **用户不在看**：标签页不可见时停止处理（`document.visibilityState === 'hidden'`）
3. **性能压力**：帧率低于阈值时，每两帧处理一帧（隔帧处理），用插值或直接重复上一帧

副作用：

- 跳帧会导致预览和最终导出效果不一致——用户在预览中看到的调色，和最终逐帧导出的结果可能有差异
- 帧差检测本身有计算开销，如果视频运动量大（如体育赛事），帧帧都不同，检测反而浪费了时间
- 用户体验上，预览跳帧会让特效看起来不连贯

实际编辑器（如 Premiere）的策略是：预览时根据性能压力动态调整分辨率和帧率，导出时始终全分辨率逐帧处理——**预览保流畅，导出保质量**。
