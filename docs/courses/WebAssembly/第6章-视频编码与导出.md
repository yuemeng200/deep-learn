# 第6章：视频编码与导出

前五章我们完成了一个能实时预览视频特效的引擎——JS 调度帧循环，Wasm 处理像素，Worker 解放主线程。但一个编辑器最关键的功能之一是**导出**：用户编辑完成后，需要把处理后的帧序列编码为可下载的视频文件。本章我们将理解视频编码的基本原理，并使用 WebCodecs API + Wasm 实现完整的导出链路。

---

## 6.1 视频编码的本质：从 33MB/帧到合理的文件大小

一段 10 秒的 1080p 30fps 视频，原始帧数据量：

```
10 秒 × 30 帧/秒 × 1920 × 1080 × 4 字节 ≈ 2.5 GB
```

2.5 GB 的原始数据——这就是视频编码要解决的问题：**在保持视觉质量的前提下，把数据量压缩到原来的 1/50 到 1/200**。

### 编码器做了什么

视频编码器的核心洞察：**相邻帧之间大部分像素是相同的。**

```
帧1（关键帧/I帧）：完整图像，独立编码
帧2（P帧）：只记录和帧1的差异（运动矢量 + 残差）
帧3（P帧）：只记录和帧2的差异
...
帧30（I帧）：新的关键帧
```

这就是 H.264/AVC、H.265/HEVC 等编码格式的基本原理。具体包含三个维度：

| 维度 | 压缩什么 | 方法 |
|------|---------|------|
| 空间冗余 | 单帧内相邻像素的相似性 | DCT 变换 + 量化 |
| 时间冗余 | 相邻帧之间的相似性 | 运动估计 + 运动补偿 |
| 统计冗余 | 编码符号的频率不均匀 | 熵编码（CABAC/CAVLC） |

### 为什么不直接用 Wasm 做编码

理论上可以，但：

1. **编码计算量巨大**：运动估计是 O(n²) 复杂度的块匹配算法，纯软件编码 1080p 可能只有 2-5fps
2. **浏览器已有硬件编码器**：现代浏览器通过 WebCodecs API 暴露了 GPU/硬件编码能力，速度比纯软件快 10-50 倍
3. **专利问题**：H.264/H.265 涉及专利许可，浏览器厂商已为此付费，自己实现反而有法律风险

所以我们的策略是：**Wasm 做像素处理，WebCodecs 做编码**——又是协作。

---

## 6.2 WebCodecs API：浏览器中的编码基础设施

WebCodecs 是 W3C 的现代 API，提供对浏览器底层编解码器的直接访问。

### 6.2.1 核心对象

```
VideoEncoder          ← 编码帧序列
VideoDecoder          ← 解码帧序列（之前用 <video> 做的事）
VideoFrame            ← 一帧图像数据（替代 getImageData）
EncodedVideoChunk     ← 编码后的数据块
```

### 6.2.2 编码流程

```
原始帧（VideoFrame / RGBA 数据）
    ↓ VideoEncoder.encode(frame)
编码器内部（硬件/软件）
    ↓ 回调 output(chunk)
编码块（EncodedVideoChunk）
    ↓ 收集所有块
写入容器（MP4/WebM）
    ↓
可下载的视频文件
```

### 6.2.3 容器格式

编码器只负责压缩视频帧数据（"裸流"），不管封装。要生成可播放的文件，还需要一个**容器格式**：

- **MP4**：最通用的容器，需要按 MP4 规范组织数据结构
- **WebM**：基于 Matroska，浏览器原生支持良好

我们可以用 `mp4-muxer` 或 `webm-muxer` 这样的 JS 库来处理容器封装。

### 步骤三：Rust 侧——添加导出用的批量处理接口

在 `src/lib.rs` 中添加：

```rust
use wasm_bindgen::prelude::*;

#[wasm_bindgen]
extern "C" {
    #[wasm_bindgen(js_namespace = console)]
    fn log(s: &str);
}

macro_rules! console_log {
    ($($t:tt)*) => (log(&format_args!($($t)*).to_string()))
}

struct FilterConfig {
    brightness: i16,
    contrast: f32,
    saturation: f32,
    grayscale: bool,
}

#[wasm_bindgen]
pub struct Pipeline {
    width: u32,
    height: u32,
    config: FilterConfig,
    frame_count: u32,
}

#[wasm_bindgen]
impl Pipeline {
    #[wasm_bindgen(constructor)]
    pub fn new(width: u32, height: u32) -> Self {
        Pipeline {
            width,
            height,
            config: FilterConfig {
                brightness: 0,
                contrast: 1.0,
                saturation: 1.0,
                grayscale: false,
            },
            frame_count: 0,
        }
    }

    pub fn set_filter(
        &mut self,
        brightness: i16,
        contrast: f32,
        saturation: f32,
        grayscale: bool,
    ) {
        self.config = FilterConfig {
            brightness,
            contrast,
            saturation,
            grayscale,
        };
    }

    /// 实时预览用：原地处理
    pub fn process_frame(&mut self, data: &mut [u8]) -> u32 {
        self.frame_count += 1;
        self.apply_all_filters(data);
        self.frame_count
    }

    /// 导出用：处理一帧并返回 RGBA 数据（用于 VideoFrame 创建）
    pub fn process_frame_for_export(&mut self, data: &[u8]) -> Vec<u8> {
        let mut output = data.to_vec();
        self.apply_all_filters(&mut output);
        self.frame_count += 1;
        output
    }

    fn apply_all_filters(&self, data: &mut [u8]) {
        let c = &self.config;
        for chunk in data.chunks_exact_mut(4) {
            let mut r = chunk[0] as f32;
            let mut g = chunk[1] as f32;
            let mut b = chunk[2] as f32;

            if c.grayscale {
                let gray = 0.299 * r + 0.587 * g + 0.114 * b;
                r = gray; g = gray; b = gray;
            }

            if c.brightness != 0 {
                let br = c.brightness as f32;
                r = (r + br).clamp(0.0, 255.0);
                g = (g + br).clamp(0.0, 255.0);
                b = (b + br).clamp(0.0, 255.0);
            }

            if c.contrast != 1.0 {
                r = ((r - 128.0) * c.contrast + 128.0).clamp(0.0, 255.0);
                g = ((g - 128.0) * c.contrast + 128.0).clamp(0.0, 255.0);
                b = ((b - 128.0) * c.contrast + 128.0).clamp(0.0, 255.0);
            }

            if c.saturation != 1.0 {
                let gray = 0.299 * r + 0.587 * g + 0.114 * b;
                r = (gray + (r - gray) * c.saturation).clamp(0.0, 255.0);
                g = (gray + (g - gray) * c.saturation).clamp(0.0, 255.0);
                b = (gray + (b - gray) * c.saturation).clamp(0.0, 255.0);
            }

            chunk[0] = r as u8;
            chunk[1] = g as u8;
            chunk[2] = b as u8;
        }
    }

    pub fn frame_count(&self) -> u32 { self.frame_count }
    pub fn reset_counter(&mut self) { self.frame_count = 0; }
    pub fn info(&self) -> String {
        format!("Pipeline: {}x{}, {} frames processed", self.width, self.height, self.frame_count)
    }
}
```

### 步骤四：JS 侧——编码导出模块

创建 `exporter.js`：

```javascript
// exporter.js — 视频导出模块
import { Muxer, ArrayBufferTarget } from 'mp4-muxer';

export class VideoExporter {
    constructor(width, height, fps = 30) {
        this.width = width;
        this.height = height;
        this.fps = fps;
        this.encodedChunks = [];
    }

    async export(videoElement, pipeline, onProgress) {
        const duration = videoElement.duration;
        const totalFrames = Math.ceil(duration * this.fps);

        // 检查 WebCodecs 支持
        if (!('VideoEncoder' in window)) {
            throw new Error('浏览器不支持 WebCodecs API，请使用 Chrome 94+ 或 Edge 94+');
        }

        // 配置 MP4 封装器
        let muxerOutput = null;
        const muxer = new Muxer({
            target: new ArrayBufferTarget(),
            video: {
                codec: 'avc',
                width: this.width,
                height: this.height,
            },
            fastStart: 'in-memory',
        });

        // 创建编码器
        const encoder = new VideoEncoder({
            output: (chunk, meta) => {
                muxer.addVideoChunk(chunk, meta);
            },
            error: (e) => {
                console.error('编码错误:', e);
            },
        });

        // 配置编码参数
        encoder.configure({
            codec: 'avc1.64001f', // H.264 High Profile
            width: this.width,
            height: this.height,
            bitrate: 8_000_000, // 8 Mbps（1080p 推荐码率）
            framerate: this.fps,
        });

        // 创建用于帧提取的离屏 Canvas
        const offscreen = document.createElement('canvas');
        offscreen.width = this.width;
        offscreen.height = this.height;
        const offCtx = offscreen.getContext('2d', { willReadFrequently: true });

        // 逐帧处理
        for (let i = 0; i < totalFrames; i++) {
            // 定位到当前帧
            const time = i / this.fps;
            await this.seekTo(videoElement, time);

            // 提取帧数据
            offCtx.drawImage(videoElement, 0, 0, this.width, this.height);
            const imageData = offCtx.getImageData(0, 0, this.width, this.height);

            // Wasm 处理
            const processed = pipeline.process_frame_for_export(imageData.data);

            // 创建 VideoFrame 并编码
            const frame = new VideoFrame(processed, {
                format: 'RGBA',
                codedWidth: this.width,
                codedHeight: this.height,
                timestamp: (i * 1_000_000) / this.fps, // 微秒
                duration: 1_000_000 / this.fps,
            });

            encoder.encode(frame);
            frame.close();

            // 报告进度
            if (onProgress) {
                onProgress(i + 1, totalFrames);
            }

            // 让浏览器有机会处理已编码的数据
            if (i % 10 === 0) {
                await encoder.flush();
                await new Promise(r => setTimeout(r, 0));
            }
        }

        // 完成编码
        await encoder.flush();
        muxer.finalize();

        // 获取 MP4 数据
        const mp4Buffer = muxer.target.buffer;

        // 清理
        encoder.close();

        return mp4Buffer;
    }

    seekTo(video, time) {
        return new Promise((resolve) => {
            video.currentTime = time;
            video.addEventListener('seeked', resolve, { once: true });
        });
    }

    // 下载 MP4 文件
    static download(buffer, filename = 'export.mp4') {
        const blob = new Blob([buffer], { type: 'video/mp4' });
        const url = URL.createObjectURL(blob);
        const a = document.createElement('a');
        a.href = url;
        a.download = filename;
        a.click();
        URL.revokeObjectURL(url);
    }
}
```

## 6.4 关键解析

### 6.4.1 VideoFrame 与 ImageData 的区别

```
ImageData: JS 对象，像素数据存在 JS 堆中
           适用于 Canvas 2D 上下文的读写
           创建/销毁由 GC 管理

VideoFrame: 浏览器原生对象，像素数据可存在 GPU 内存中
            适用于 WebCodecs 编解码
            必须 .close() 手动释放
```

`new VideoFrame(uint8Array, options)` 接受 RGBA 数据并创建帧对象。帧的 `timestamp` 以微秒为单位，这是编解码器的标准时间单位。

### 6.4.2 编码器配置参数

```javascript
encoder.configure({
    codec: 'avc1.64001f',   // H.264 编码器标识
    bitrate: 8_000_000,     // 目标码率（比特/秒）
    framerate: 30,          // 帧率
});
```

`avc1.64001f` 解读：
- `avc1`：H.264/AVC 编码
- `64`：High Profile
- `001f`：Level 3.1（支持 1080p 30fps）

码率选择参考：
| 分辨率 | 推荐码率 |
|--------|---------|
| 720p | 2-5 Mbps |
| 1080p | 5-10 Mbps |
| 4K | 20-50 Mbps |

### 6.4.3 导出 vs 预览的架构差异

```
预览（实时）：
<video> → Canvas → getImageData → Wasm → putImageData → 显示
                 ↑ 这里是瓶颈（GPU→CPU→GPU 往返）

导出（离线）：
<video> seek → Canvas → getImageData → Wasm → VideoFrame → VideoEncoder → MP4
             ↑ 不需要实时，逐帧处理，追求质量
```

导出时不需要 `requestAnimationFrame`，逐帧 seek→处理→编码，速度取决于编码器。硬件编码器通常能做到 60-120fps（比实时快 2-4 倍），但 seek 操作有 I/O 延迟，实际速度约 15-30fps。

## 6.5 WebCodecs 的兼容性与回退

WebCodecs 是较新的 API（Chrome 94+，2021 年发布）。对于不支持 WebCodecs 的浏览器：

### 方案一：降级为 GIF 导出

```javascript
// 使用 gif.js 库导出 GIF（纯 JS 编码，兼容性好）
import GIF from 'gif.js';

async function exportGif(frames, width, height) {
    const gif = new GIF({
        workers: 2,
        quality: 10,
        width,
        height,
    });

    for (const frame of frames) {
        gif.addFrame(frame, { delay: 1000 / 30 });
    }

    return new Promise(resolve => {
        gif.on('finished', blob => resolve(blob));
        gif.render();
    });
}
```

### 方案二：使用 FFmpeg.wasm

```javascript
// 使用 Wasm 版 FFmpeg 做编码（体积大但功能全）
import { FFmpeg } from '@ffmpeg/ffmpeg';

const ffmpeg = new FFmpeg();
await ffmpeg.load();

// 把帧序列写入虚拟文件系统
for (let i = 0; i < frames.length; i++) {
    await ffmpeg.writeFile(`frame_${i.toString().padStart(5, '0')}.rgba`, frames[i]);
}

// 编码
await ffmpeg.exec([
    '-framerate', '30',
    '-i', 'frame_%05d.rgba',
    '-s', `${width}x${height}`,
    '-pix_fmt', 'rgba',
    '-c:v', 'libx264',
    '-preset', 'fast',
    'output.mp4'
]);
```

FFmpeg.wasm 体积约 30MB，加载慢但兼容所有浏览器。

## 章末思考题

**题目一**：导出时，我们用 `video.currentTime = time` 然后 `await seeked` 来逐帧定位。这个过程涉及视频解码——浏览器需要解码目标帧附近的 I 帧，然后逐帧解码到目标帧。如果视频是 30fps 的 10 分钟视频（18000 帧），seek 18000 次大约需要多久？有没有更高效的方式？

**参考答案**：

每次 seek 的延迟约 10-50ms（取决于视频编码复杂度和是否需要跨 I 帧组）。18000 次请求大约需要 3-15 分钟。

更高效的方式是**顺序播放而非随机请求**：

```javascript
// 顺序模式：从头播放视频，每帧都处理
video.playbackRate = 0; // 暂停播放
video.currentTime = 0;
await seekTo(video, 0);

for (let i = 0; i < totalFrames; i++) {
    // 小步前进（相邻帧，无需重新解码）
    video.currentTime = (i + 0.5) / fps;
    await seekTo(video, video.currentTime);
    // ... 处理帧 ...
}
```

更好的方案是使用 **VideoDecoder API** 直接解码帧序列，完全绕过 `<video>` 元素。VideoDecoder 可以一次性送入压缩的视频块，按顺序输出解码后的帧——避免反复请求的开销。

---

**题目二**：我们选择了 H.264 编码（`avc1.64001f`）。H.264 是 2003 年发布的编码标准。更新的 H.265/HEVC 和 AV1 编码器压缩率更高（同等质量下体积小 40-50%）。为什么仍然推荐 H.264？

**参考答案**：

**兼容性**。H.264 是目前支持最广泛的编码格式——几乎所有设备、操作系统、浏览器都支持 H.264 硬件解码。H.265 的浏览器支持有限（Safari 原生支持，Chrome/Escargot 需要 OS 级解码器），AV1 的硬件解码器在旧设备上不可用。

在视频编辑器场景中，导出的视频可能需要分享给其他人——选择 H.264 确保对方能播放。如果确定目标平台支持，可以在 `configure()` 中切换编码器：

```javascript
// AV1（体积最小，编码最慢）
{ codec: 'av01.0.08M.08' }

// H.265（体积中等，兼容性较差）
{ codec: 'hev1.1.6.L93.B0' }

// H.264（体积最大，兼容性最好）
{ codec: 'avc1.64001f' }
```

实际项目中，可以提供"快速导出（H.264）"和"高质量导出（AV1）"两个选项。

---

**题目三**：`VideoFrame` 对象必须手动 `.close()` 释放。如果不调用 close 会怎样？这和 JS 的垃圾回收有什么关系？

**参考答案**：

`VideoFrame` 持有外部资源（可能是 GPU 纹理内存、共享内存缓冲区等），这些资源不受 JS 垃圾回收管理。如果不 close：

1. **内存泄漏**：每帧约 8MB，30fps 编码 1 分钟视频 = 1800 帧 = 14.4 GB 未释放的帧数据。浏览器会在某个时刻崩溃或变得极慢。
2. **编码器阻塞**：部分浏览器的 VideoEncoder 实现有内部帧缓冲区限制（通常 5-10 帧），如果不释放已编码的帧，编码器会在 `encode()` 时阻塞等待。

这和 Rust 的所有权模型形成了有趣的对比——Rust 在编译时强制你管理资源生命周期，而 WebCodecs API 在运行时依赖你手动 close。这是两个世界（Wasm 和 Web API）在设计哲学上的差异。
