# 第7章：综合实战 — 从报警到定位的完整排查流程

> 前六章你分别学了日志地图、基础命令、文本处理、实时追踪、日志规范、管道组合。本章把所有武器装上膛，模拟一次**完整的线上故障排查**——从收到报警到定位根因、验证修复。这是"向正确的数据提出正确的问题"这一核心能力的完整演练。

---

## 7.1 排查方法论：SOAR 四步框架

```
S — Scope（定范围）    影响了什么？多大范围？什么时候开始的？
O — Observe（看现场）  当前系统状态是什么？日志说了什么？
A — Analyze（找根因）  从现象出发，逐层排除，锁定根因
R — Resolve（解决+复盘）修复问题，验证恢复，记录经验
```

---

## 7.2 实战场景：订单服务报警

### 背景

你是 `order-service` 的开发者，Go 写的 HTTP 服务，CentOS + systemd 管理，slog 输出 JSON 到 stdout。

**周三下午 14:30，收到报警**：order-service 错误率超过 5%，持续 3 分钟。

---

### Step 1：Scope — 定范围

```bash
# 1. 服务还活着吗？
systemctl status order-service

# 2. 错误什么时候开始的？
journalctl -u order-service --since "30 min ago" -o cat --no-pager \
  | jq -r 'select(.level=="ERROR") | .time[:16]' \
  | sort | uniq -c

# 输出显示 14:25 开始暴增

# 3. 是所有接口还是某个接口？
journalctl -u order-service --since "14:25" -o cat --no-pager \
  | jq -r 'select(.level=="ERROR") | .path' \
  | sort | uniq -c | sort -rn

# 结论：主要是 /api/order 接口
```

### Step 2：Observe — 看现场

```bash
# 4. 错误类型统计
journalctl -u order-service --since "14:25" -o cat --no-pager \
  | jq -r 'select(.level=="ERROR" and .path=="/api/order") | .err' \
  | sort | uniq -c | sort -rn

# 发现：几乎全是 MySQL connection refused / timed out

# 5. 还有成功请求吗？
journalctl -u order-service --since "14:25" -o cat --no-pager \
  | jq -r 'select(.path=="/api/order") | .level' \
  | sort | uniq -c

# 部分请求仍成功——MySQL 不是完全挂了

# 6. 超时耗时
journalctl -u order-service --since "14:25" -o cat --no-pager \
  | jq 'select(.level=="ERROR" and .path=="/api/order") | .latency_ms' \
  | sort -rn | head -n 5

# 约 5 秒——符合 MySQL 连接超时配置
```

### Step 3：Analyze — 找根因

```bash
# 7. MySQL 状态
ssh 10.0.1.50 "systemctl status mysqld"
ssh 10.0.1.50 "journalctl -u mysqld --since '14:20' --no-pager | tail -n 30"

# 发现：Too many connections

# 8. 流量是否突增？
journalctl -u order-service --since "14:00" --until "14:30" -o cat --no-pager \
  | jq -r '.time[:16]' \
  | sort | uniq -c

# 14:20 请求量暴增 16 倍！

# 9. 谁发的？
journalctl -u order-service --since "14:20" --until "14:25" -o cat --no-pager \
  | jq -r '.uid' \
  | sort | uniq -c | sort -rn | head -n 10

# 根因：batch-import-001 发起了 612 个并发请求，打满 MySQL 连接池
```

### Step 4：Resolve — 解决 + 复盘

```bash
# 临时止血
ssh 10.0.1.50 "mysql -e 'SET GLOBAL max_connections = 500;'"

# 验证恢复
journalctl -u order-service -f -o cat | jq --unbuffered 'select(.level=="ERROR") | [.time, .err] | @tsv'

# 最近 5 分钟错误数
journalctl -u order-service --since "5 min ago" -o cat --no-pager \
  | jq 'select(.level=="ERROR")' | wc -l

# 导出故障数据
journalctl -u order-service --since "14:20" --until "14:40" -o cat --no-pager \
  | jq 'select(.level=="ERROR")' \
  > /tmp/incident-2024-03-15-errors.json
```

## 7.3 排查命令速查——按阶段整理

```
阶段              命令                                    目的
──────────────────────────────────────────────────────────────────
【定范围】
服务是否存活      systemctl status SERVICE                快速概览
错误时间分布      ... | jq '.time[:16]' | sort | uniq -c  确定起始时间
影响哪些接口      ... | jq '.path' | sort | uniq -c       缩小范围

【看现场】
错误类型统计      ... | jq '.err' | sort | uniq -c        分类错误
成功失败比例      ... | jq '.level' | sort | uniq -c      判断严重程度
超时耗时分布      ... | jq '.latency_ms' | sort -rn       确认超时模式

【找根因】
流量是否突增      ... | jq '.time[:16]' | sort | uniq -c  看趋势
谁发的请求        ... | jq '.uid' | sort | uniq -c        定位来源
系统层面检查      dmesg -T | grep oom                      排除 OOM
上游/下游日志     ssh HOST "journalctl ..."                跨服务排查

【验证恢复】
实时看错误        journalctl -f -o cat | jq '...'          确认修复
最近错误数        ... --since "5 min ago" | ... | wc -l    量化恢复
导出故障数据      ... | jq '...' > incident.json            留档复盘
```

## 7.4 排查直觉清单

| 症状 | 第一反应 |
|------|---------|
| 服务没了，应用日志无异常 | `dmesg -T \| grep oom` |
| 服务启动失败 | `journalctl -u SERVICE -b --no-pager \| tail -30` |
| 错误率突增 | 先按接口分类，再看错误类型，再看流量趋势 |
| 响应变慢但没报错 | 找 latency 分布，看是否有慢查询/外部依赖超时 |
| 磁盘满了 | `du -sh /var/log/* \| sort -rh \| head` |
| SSH 不上去 | 先 ping，再看 /var/log/secure |
| 日志突然不写了 | 检查磁盘空间、文件句柄数 |
| 定时任务没跑 | /var/log/cron 或 journalctl -u crond |
| 重启后日志丢了 | 检查 /var/log/journal/ 是否存在 |

## 章末思考题

**题目一：在上面的实战中，我们从应用日志查到了 MySQL。但如果是微服务架构，order-service 调用了 user-service 和 payment-service，三个服务日志在不同机器。你要追踪一个请求的完整链路，只靠 grep uid 可以吗？有什么更好的方案？**

> **参考答案**：只靠 uid 不够可靠——同一个 uid 可能同时有多个并发请求，无法区分。更好的方案是**分布式链路追踪**：在入口服务生成全局唯一的 `trace_id`，随请求传递到每个下游服务，每条日志都带上 trace_id。排查时 `grep "trace_id=xxx"` 就能跨服务串联完整链路。Go 中可以用 OpenTelemetry 实现。这是从"单机日志排查"到"分布式可观测性"的自然演进。

**题目二：实战中用 `SET GLOBAL max_connections = 500` 临时增大了 MySQL 连接数。这是一个正确的"修复"吗？真正应该做什么？**

> **参考答案**：这是**止血**，不是修复。真正要解决两层问题：① **应用层**：batch-import-001 的批量请求应该限流限速（`golang.org/x/time/rate` 令牌桶或信号量控制并发）；② **连接层**：Go 服务的数据库连接池应设置合理的 `MaxOpenConns` 和 `MaxIdleConns`，防止一个服务吃光 MySQL 连接。止血是必要的，但必须跟进根本性修复。

**题目三：回顾整个排查过程，如果让你写一个 shell 脚本自动完成"收集故障现场信息"，你会收集哪些数据？**

> **参考答案**：
> ```bash
> #!/bin/bash
> SERVICE=$1
> SINCE=${2:-"30 min ago"}
> OUT="/tmp/incident-$(date +%Y%m%d-%H%M%S)"
> mkdir -p $OUT
>
> systemctl status $SERVICE > $OUT/service-status.txt 2>&1
> journalctl -u $SERVICE --since "$SINCE" -o cat --no-pager > $OUT/raw-logs.json
> cat $OUT/raw-logs.json | jq -r 'select(.level=="ERROR") | .err' | sort | uniq -c | sort -rn > $OUT/error-summary.txt
> cat $OUT/raw-logs.json | jq -r '.time[:16]' | sort | uniq -c > $OUT/traffic-trend.txt
> dmesg -T | tail -n 50 > $OUT/dmesg.txt
> free -h > $OUT/memory.txt
> df -h > $OUT/disk.txt
>
> echo "现场数据已保存到 $OUT/"
> ```
> 30 秒收集最关键的排查数据。在真实团队中，这类脚本叫"runbook"，是运维自动化的起点。
