# 第1章：日志的世界观 — Linux 日志体系全景

> 我们说日志排查的本质是"向正确的数据提出正确的问题"。那第一个问题自然是——**数据在哪？** 本章帮你建立一张完整的"日志地图"，以后出了问题，你至少知道该去哪里找。

---

## 1.1 日志到底是什么

先抛开技术，想一个类比：日志就是系统的**监控录像**。

你开了一家餐厅，装了三种摄像头：
- **门口摄像头**：记录谁进来了、谁出去了（→ 访问日志）
- **厨房摄像头**：记录厨师做了什么操作（→ 应用业务日志）
- **电路监控**：记录电压、跳闸、设备故障（→ 系统/内核日志）

出了问题时，你不会把所有录像从头看一遍，而是**根据问题类型去看对应的摄像头**。日志排查也一样。

在 Linux 上，日志本质就是**不断追加内容的文本文件**（或二进制 journal）。程序把发生的事情一行行写进去，我们事后来读。

---

## 1.2 Linux 日志体系：三大类

在你的 CentOS 服务器上，日志分为三大类：

### ① 系统日志（由 systemd/rsyslog 管理）

这是操作系统自己记录的"大事记"，住在 `/var/log/` 下面：

```
/var/log/
├── messages      # 最重要！系统综合日志，大部分系统事件都在这里
├── secure        # 登录、sudo、SSH 相关（谁登了你的服务器？）
├── cron          # 定时任务执行记录
├── dmesg         # 内核启动时的硬件/驱动信息
├── boot.log      # 系统启动过程日志
├── yum.log       # 软件安装/更新记录
└── wtmp / btmp   # 登录成功/失败记录（二进制，用 last/lastb 读）
```

**CentOS 7+ 特别说明**：CentOS 7 开始使用 systemd，它有自己的日志系统叫 **journald**，用 `journalctl` 命令查看。journald 收集了所有由 systemd 管理的服务的日志（包括你的 Go 服务，如果你用 systemd 管理它的话）。

`/var/log/messages` 和 `journalctl` 的关系：journald 收集日志后，rsyslog 会把一部分转存到 `/var/log/messages`。两者内容有重叠，但 journalctl 更全、支持更灵活的查询。

### ② 应用日志（由你的程序自己写）

这是你的 Go 服务自己产生的日志。它住在哪里，**完全取决于你怎么配置**。常见位置：

```
# 方式一：写到文件
/var/log/myapp/app.log          # 自定义目录
/opt/myapp/logs/app.log         # 跟着应用走

# 方式二：输出到 stdout/stderr
# 如果用 systemd 管理，会被 journald 收集
# 如果用 Docker 运行，会被 docker logs 收集

# 方式三：直接发送到远程（ELK、Loki 等）
# 本地没有文件，要去日志平台查
```

**在 Go 项目中**，最常见的做法是输出到 stdout，然后让外层（systemd / Docker / k8s）去处理日志的收集和存储。

### ③ 内核日志

内核遇到的严重事件：硬件故障、内存不足（OOM）、文件系统错误等。

```
dmesg                          # 查看内核环形缓冲区
/var/log/dmesg                 # 启动时的内核日志快照
/var/log/messages              # 也会包含部分内核日志
```

当你的 Go 服务被系统强杀（OOM Killer），原因不在应用日志里，而在内核日志里——这是很多人排查不出问题的原因。

## 1.3 实战：画出你的日志地图

假设我们的 Go 订单服务 `order-service` 跑在 CentOS 上，用 systemd 管理。出了问题时该查哪里？

```
问题类型                    →  去哪查
─────────────────────────────────────────────
接口返回 500 错误           →  应用日志（Go 程序输出）
服务启动失败                →  journalctl -u order-service
服务突然消失、被杀死        →  dmesg | grep -i oom
                               journalctl -u order-service
SSH 登不上服务器            →  /var/log/secure
定时任务没执行              →  /var/log/cron
磁盘满了导致写日志失败      →  df -h 看磁盘 + /var/log/messages
依赖的 MySQL 连接不上       →  应用日志 + /var/log/messages（网络相关）
服务器重启了但不知道为什么   →  last -x（重启记录）+ /var/log/messages
```

这张表就是你的**日志地图**。每次遇到问题，先花 3 秒钟判断"这属于哪一类"，然后直奔对应的日志。不要上来就 `grep` 满世界搜——那是大海捞针。

## 1.4 一个关键认知：日志的层次感

```
        ┌─────────────────────────┐
        │    应用日志（Go 程序）    │  ← 业务细节最丰富，但视野最窄
        ├─────────────────────────┤
        │  服务管理日志（systemd）  │  ← 能看到启停、崩溃、重启
        ├─────────────────────────┤
        │    系统日志（rsyslog）    │  ← 能看到系统级事件
        ├─────────────────────────┤
        │    内核日志（dmesg）      │  ← 最底层，OOM/硬件/驱动
        └─────────────────────────┘
```

排查问题时，**一般从上往下查**：先看应用日志（因为信息最具体），如果应用日志里找不到答案，就往下一层看。就像调试代码时，先看业务逻辑，再看框架，再看运行时。

## 章末思考题

**题目一：你的 Go 服务跑着跑着突然没了，应用日志的最后一行是正常的业务日志，没有任何 error 或 panic。这时候你该往哪个方向查？为什么？**

> **参考答案**：应该往系统层查，优先看 `dmesg | grep -i oom` 和 `journalctl -u your-service`。应用日志"正常结束"说明程序不是自己崩的，最常见的原因是被 OOM Killer 强制杀死——内核发现内存不足时会选一个"占内存最多的进程"直接 kill，程序来不及写任何日志。这就是为什么日志有"层次"——应用层看不到的答案，要去内核层找。

**题目二：`/var/log/messages` 和 `journalctl` 都能看到系统日志，实际排查时你会优先用哪个？**

> **参考答案**：优先用 `journalctl`。原因有三：① 它能按服务过滤（`journalctl -u order-service`），而 messages 文件混杂了所有服务的日志；② 它支持时间范围查询（`--since "2024-01-01 10:00"`），不需要自己 grep 时间；③ 它包含的信息更全（比如进程 PID、优先级等元数据）。`/var/log/messages` 更适合在 journald 不可用时作为备选，或者快速用 `tail` 看最近发生了什么。

**题目三：为什么推荐 Go 服务把日志输出到 stdout 而不是直接写文件？**

> **参考答案**：这遵循"Twelve-Factor App"的日志原则——应用只负责产生日志流，不关心日志存到哪里。好处是：① 让运行环境（systemd/Docker/k8s）统一管理日志的收集、轮转和存储，避免每个应用各自为政；② stdout 的日志会被 journald 或 Docker 自动收集，天然具备 `journalctl` 查询能力；③ 应用代码更简单，不需要处理文件打开、轮转、磁盘满等问题。但在一些高吞吐场景下，直接写文件可以减少中间层开销，这是一个权衡。
