# 第4章：会话管理与访问限流

> 前三章我们用 Redis 存数据、做缓存，都是在"读写加速"的维度上发力。本章我们将看到 Redis 的另一面——**用原子操作和过期机制实现控制逻辑**。会话管理和限流的核心都不是"存什么数据"，而是"在并发环境下精确地控制状态"，这正是 Redis 单线程原子性的用武之地。

---

## 4.1 会话管理：为什么不用 Cookie 直接存？

Web 应用需要记住"这个请求是谁发的"。传统做法有两种：

```
方案 A：Cookie 存全部信息（JWT / 编码后的用户数据）
  客户端 ←→ 服务端
  ✓ 服务端无状态，容易水平扩展
  ✗ 无法主动让某个会话失效（比如踢人下线）
  ✗ Cookie 体积大，每次请求都带着

方案 B：Cookie 只存 Session ID，数据存服务端
  客户端 ──SessionID──→ 服务端 ──查 Redis──→ 会话数据
  ✓ 可以主动销毁会话（删 Redis key 即可）
  ✓ Cookie 体积极小
  ✗ 服务端需要一个共享存储
```

Redis 天然适合方案 B：内存快、支持过期、多个服务实例可以共享同一个 Redis。

### 心智模型

```
用户登录成功
    │
    ▼
生成随机 Session ID: "sess:a3f8b2c1..."
    │
    ▼
Redis: SET "sess:a3f8b2c1..." → { userID, name, role, ... }  TTL=24h
    │
    ▼
将 Session ID 写入 Cookie 返回给客户端

后续每次请求
    │
    ▼
从 Cookie 取 Session ID → 查 Redis → 得到用户身份
```

## 4.2 实现会话管理

```go
package main

import (
    "context"
    "crypto/rand"
    "encoding/hex"
    "encoding/json"
    "errors"
    "fmt"
    "log"
    "net/http"
    "time"

    "github.com/redis/go-redis/v9"
)

var rdb *redis.Client

func initRedis() {
    rdb = redis.NewClient(&redis.Options{Addr: "localhost:6379"})
    if err := rdb.Ping(context.Background()).Err(); err != nil {
        log.Fatal(err)
    }
}

// ---------- 会话模块 ----------

const (
    sessionPrefix = "sess:"
    sessionTTL    = 24 * time.Hour
    cookieName    = "session_id"
)

type Session struct {
    UserID   int64  `json:"user_id"`
    UserName string `json:"user_name"`
    Role     string `json:"role"`
}

// 生成安全的随机 Session ID
func generateSessionID() string {
    b := make([]byte, 32) // 256 位随机数
    rand.Read(b)
    return hex.EncodeToString(b)
}

func sessionKey(id string) string {
    return sessionPrefix + id
}

// 创建会话（登录成功后调用）
func CreateSession(ctx context.Context, sess *Session) (string, error) {
    sid := generateSessionID()

    data, _ := json.Marshal(sess)
    err := rdb.Set(ctx, sessionKey(sid), data, sessionTTL).Err()
    if err != nil {
        return "", fmt.Errorf("创建会话失败: %w", err)
    }

    return sid, nil
}

// 获取会话
func GetSession(ctx context.Context, sid string) (*Session, error) {
    data, err := rdb.Get(ctx, sessionKey(sid)).Result()
    if errors.Is(err, redis.Nil) {
        return nil, nil // 会话不存在或已过期
    }
    if err != nil {
        return nil, err
    }

    var sess Session
    json.Unmarshal([]byte(data), &sess)
    return &sess, nil
}

// 销毁会话（退出登录 / 踢人下线）
func DestroySession(ctx context.Context, sid string) error {
    return rdb.Del(ctx, sessionKey(sid)).Err()
}

// 续期 —— 用户活跃时延长会话有效期
func RefreshSession(ctx context.Context, sid string) error {
    return rdb.Expire(ctx, sessionKey(sid), sessionTTL).Err()
}
```

**要点**：
- **`crypto/rand`** 而非 `math/rand`：Session ID 是安全凭证，必须用密码学安全的随机数生成器
- **TTL 即会话过期**：不需要额外的定时清理任务，Redis 的过期机制自动回收
- **`Expire` 续期**：用户每次活跃时刷新 TTL，实现"24小时无操作才过期"的滑动窗口效果

## 4.3 会话中间件

把会话校验抽成 HTTP 中间件，所有需要登录的接口都自动获得身份信息：

```go
// 中间件：从请求中提取会话，注入到 context
func sessionMiddleware(next http.HandlerFunc) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        cookie, err := r.Cookie(cookieName)
        if err != nil {
            http.Error(w, "未登录", http.StatusUnauthorized)
            return
        }

        sess, err := GetSession(r.Context(), cookie.Value)
        if err != nil {
            http.Error(w, "服务内部错误", http.StatusInternalServerError)
            return
        }
        if sess == nil {
            http.Error(w, "会话已过期，请重新登录", http.StatusUnauthorized)
            return
        }

        // 续期：用户活跃则延长会话
        RefreshSession(r.Context(), cookie.Value)

        // 将用户信息注入 context，下游 handler 可以直接使用
        ctx := context.WithValue(r.Context(), "session", sess)
        next(w, r.WithContext(ctx))
    }
}

// 登录接口
func loginHandler(w http.ResponseWriter, r *http.Request) {
    // 实际项目中这里要验证用户名密码
    sess := &Session{
        UserID:   1001,
        UserName: "张三",
        Role:     "user",
    }

    sid, err := CreateSession(r.Context(), sess)
    if err != nil {
        http.Error(w, "登录失败", http.StatusInternalServerError)
        return
    }

    http.SetCookie(w, &http.Cookie{
        Name:     cookieName,
        Value:    sid,
        Path:     "/",
        HttpOnly: true, // 防止 JS 读取，抵御 XSS
        MaxAge:   86400,
    })

    w.Write([]byte("登录成功"))
}

// 需要登录的接口示例
func profileHandler(w http.ResponseWriter, r *http.Request) {
    sess := r.Context().Value("session").(*Session)
    json.NewEncoder(w).Encode(map[string]interface{}{
        "user_id": sess.UserID,
        "name":    sess.UserName,
    })
}
```

## 4.4 访问限流：保护你的 API

限流解决的问题：防止单个用户/IP 在短时间内发送过多请求，保护后端服务不被打垮（无论是恶意攻击还是代码 bug 导致的重试风暴）。

### 方案一：固定窗口计数器

最简单的思路——"每分钟最多 100 次请求"：

```go
// 固定窗口限流
func fixedWindowLimit(ctx context.Context, key string, limit int64, window time.Duration) (bool, error) {
    current, err := rdb.Incr(ctx, key).Result()
    if err != nil {
        return false, err
    }

    // 第一次请求时设置过期时间
    if current == 1 {
        rdb.Expire(ctx, key, window)
    }

    return current <= limit, nil
}
```

**问题**：固定窗口有"边界突刺"——如果用户在第 59 秒发了 100 次，第 61 秒（新窗口）又发了 100 次，实际上 2 秒内发了 200 次请求。

### 方案二：滑动窗口（推荐）

用 Sorted Set 实现真正的滑动窗口，精确控制"任意连续 N 秒内"的请求数：

```go
// 滑动窗口限流
func slidingWindowLimit(ctx context.Context, userID string, limit int64, window time.Duration) (bool, int64, error) {
    key := fmt.Sprintf("ratelimit:%s", userID)
    now := time.Now()
    windowStart := now.Add(-window)

    // 用 Pipeline 将多个命令打包发送，减少网络往返
    pipe := rdb.Pipeline()

    // 1. 移除窗口之外的旧记录
    pipe.ZRemRangeByScore(ctx, key, "0", fmt.Sprintf("%d", windowStart.UnixMicro()))

    // 2. 添加当前请求（用时间戳做 score，保证唯一性用微秒）
    pipe.ZAdd(ctx, key, redis.Z{
        Score:  float64(now.UnixMicro()),
        Member: fmt.Sprintf("%d", now.UnixNano()), // member 必须唯一
    })

    // 3. 统计窗口内的请求数
    countCmd := pipe.ZCard(ctx, key)

    // 4. 设置 key 的过期时间（防止冷用户的 key 永远不被清理）
    pipe.Expire(ctx, key, window)

    // 执行 Pipeline
    _, err := pipe.Exec(ctx)
    if err != nil {
        return false, 0, err
    }

    count := countCmd.Val()
    return count <= limit, count, nil
}
```

**原理图**：

```
时间轴 ──────────────────────────────────────────→
         │←────── 1分钟窗口 ──────→│
         ┊                         ┊
    旧请求（被清除）    窗口内的请求     当前请求
    ×  ×  ×         ·  ·  ·  ·  ·    ★
                    ↑ 统计这些 ↑      ↑ 新加入

    如果窗口内数量 > limit → 拒绝
```

## 4.5 限流中间件

把限流逻辑包装成中间件：

```go
func rateLimitMiddleware(limit int64, window time.Duration) func(http.HandlerFunc) http.HandlerFunc {
    return func(next http.HandlerFunc) http.HandlerFunc {
        return func(w http.ResponseWriter, r *http.Request) {
            // 用客户端 IP 作为限流维度（也可以用 userID）
            clientIP := r.RemoteAddr

            allowed, count, err := slidingWindowLimit(r.Context(), clientIP, limit, window)
            if err != nil {
                // Redis 故障时放行（降级策略：宁可放过，不能误杀）
                log.Printf("[RateLimit] Redis 错误，放行: %v", err)
                next(w, r)
                return
            }

            // 设置响应头，让客户端知道限流状态
            w.Header().Set("X-RateLimit-Limit", fmt.Sprintf("%d", limit))
            w.Header().Set("X-RateLimit-Remaining", fmt.Sprintf("%d", max(0, limit-count)))

            if !allowed {
                w.Header().Set("Retry-After", fmt.Sprintf("%d", int(window.Seconds())))
                http.Error(w, "请求过于频繁，请稍后再试", http.StatusTooManyRequests)
                return
            }

            next(w, r)
        }
    }
}
```

## 4.6 整合：完整的路由注册

```go
func main() {
    initRedis()

    // 每分钟最多 60 次请求
    limiter := rateLimitMiddleware(60, time.Minute)

    // 公开接口（限流）
    http.HandleFunc("/login", limiter(loginHandler))
    http.HandleFunc("/health", limiter(healthHandler))

    // 需要登录的接口（限流 + 会话校验）
    http.HandleFunc("/profile", limiter(sessionMiddleware(profileHandler)))
    http.HandleFunc("/post/create", limiter(sessionMiddleware(createPostHandler)))

    log.Println("服务启动于 :8080")
    log.Fatal(http.ListenAndServe(":8080", nil))
}
```

中间件的组合方式是 `limiter(sessionMiddleware(handler))`——先限流，再校验会话。顺序很重要：如果先校验会话，恶意用户可以用无效 session 绕过限流大量请求，白白消耗 Redis 查询。

## 章末思考题

**题目一：我们在限流中间件中，当 Redis 故障时选择了"放行"。如果改成"拒绝所有请求"，会怎样？哪种策略更好？**

> **参考解析**：拒绝所有请求意味着 Redis 一旦故障，整个服务不可用——限流这个辅助功能反而变成了单点故障。通常的原则是：**限流是保护措施，不应该成为可用性的拦路虎**。放行（开放降级）在 Redis 短暂故障时维持服务可用，代价是短时间失去限流保护。哪种更好取决于业务：金融交易场景可能宁可拒绝也不能超限；普通内容社区放行更合理。关键是**做出有意识的选择**，而不是无意中引入故障。

**题目二：滑动窗口限流用了 Pipeline 打包四个命令。如果不用 Pipeline 而是逐条执行，除了性能差异外，还有什么问题？**

> **参考解析**：正确性问题。四个命令逐条执行时，在 ZREMRANGEBYSCORE（清理旧记录）和 ZCARD（统计数量）之间，可能有其他请求插入了新记录，导致统计不准。Pipeline 虽然不像 MULTI/EXEC 那样保证原子性，但它把命令打包发送，减少了命令之间被插入的窗口。如果要做到严格精确，应该用 Lua 脚本将这四步合并为一个原子操作——但对于限流场景，偶尔差一两个请求通常可以接受，Pipeline 已经够用。

**题目三：会话续期（每次请求都调 `Expire`）意味着活跃用户的会话永远不会过期。这有什么安全风险？你会怎么设计一个更安全的方案？**

> **参考解析**：如果 Session ID 被窃取（比如 XSS 攻击拿到了 Cookie），攻击者只要持续使用，会话就永远有效，用户换密码都没用。改进方案：设置一个**绝对过期时间**（比如 7 天），不管多活跃到期必须重新登录。实现方式：在 Session 数据中额外存一个 `created_at` 字段，每次读取会话时检查是否超过绝对有效期。另一个补充措施是维护一个 `user:{id}:sessions` 的 Set，用户改密码时遍历并删除所有会话（强制全部下线）。
