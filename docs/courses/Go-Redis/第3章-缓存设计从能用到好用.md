# 第3章：缓存设计 — 从能用到好用

> 前两章我们掌握了 Redis 的连接方式和五大数据结构。但 Redis 被使用最多的场景，其实是**缓存**——这也是"用内存换时间"这一核心本质最直接的体现。本章我们将从"能用"走向"好用"，理解缓存设计中那些看似简单、实则暗藏杀机的问题。

---

## 3.1 先理解缓存的本质角色

缓存不是数据库的替代品，而是**数据库的挡箭牌**。

```
客户端请求
    │
    ▼
┌─────────┐  命中   ┌─────────┐
│  缓存    │ ──────→ │  返回    │   ← 快路径（微秒级）
└─────────┘         └─────────┘
    │ 未命中
    ▼
┌─────────┐         ┌─────────┐
│  数据库  │ ──────→ │  返回    │   ← 慢路径（毫秒级）
└─────────┘         └─────────┘
    │
    ▼
  回填缓存（下次就走快路径了）
```

这个模式有个正式名字：**Cache-Aside（旁路缓存）**，是最常用的缓存模式。它的规则很简单：

| 操作 | 步骤 |
|------|------|
| **读** | 先查缓存 → 命中则返回 → 未命中则查数据库 → 回填缓存 → 返回 |
| **写** | 先更新数据库 → 再删除缓存（不是更新缓存！） |

为什么写操作是"删除缓存"而不是"更新缓存"？因为更新缓存在并发场景下容易导致数据不一致——两个并发写请求，数据库写入顺序是 A→B，但缓存更新顺序可能变成 B→A，缓存就存了旧值。删除更安全：下次读的时候自然会从数据库加载最新值。

## 3.2 为帖子详情加缓存

我们假设帖子的完整内容存在 MySQL（或其他持久化存储）中，Redis 只做缓存层。这里用一个模拟的 `fetchPostFromDB` 代替真实数据库查询：

```go
package main

import (
    "context"
    "encoding/json"
    "errors"
    "fmt"
    "log"
    "time"

    "github.com/redis/go-redis/v9"
)

var rdb *redis.Client
var ctx = context.Background()

func initRedis() {
    rdb = redis.NewClient(&redis.Options{Addr: "localhost:6379"})
    if err := rdb.Ping(ctx).Err(); err != nil {
        log.Fatal(err)
    }
}

// ---------- 帖子模型 ----------

type Post struct {
    ID      int64    `json:"id"`
    Title   string   `json:"title"`
    Content string   `json:"content"`
    Author  string   `json:"author"`
    Tags    []string `json:"tags"`
}

func postCacheKey(id int64) string {
    return fmt.Sprintf("cache:post:%d", id)
}

// 模拟数据库查询（实际项目中替换为真实 DB 调用）
func fetchPostFromDB(id int64) (*Post, error) {
    log.Printf("[DB] 查询帖子 %d（慢路径）", id)
    // 模拟查询耗时
    time.Sleep(50 * time.Millisecond)

    // 模拟：ID 大于 10000 的帖子不存在
    if id > 10000 {
        return nil, nil // 帖子不存在
    }

    return &Post{
        ID:      id,
        Title:   fmt.Sprintf("帖子 %d", id),
        Content: "这是帖子内容...",
        Author:  "张三",
        Tags:    []string{"golang", "redis"},
    }, nil
}
```

### Cache-Aside 读取实现

```go
// 获取帖子 —— Cache-Aside 模式
func GetPost(id int64) (*Post, error) {
    key := postCacheKey(id)

    // 第一步：查缓存
    data, err := rdb.Get(ctx, key).Result()
    if err == nil {
        // 缓存命中
        var post Post
        if err := json.Unmarshal([]byte(data), &post); err != nil {
            return nil, fmt.Errorf("缓存数据损坏: %w", err)
        }
        log.Printf("[Cache] 命中帖子 %d", id)
        return &post, nil
    }

    // 如果不是"key 不存在"的错误，说明 Redis 出了其他问题
    if !errors.Is(err, redis.Nil) {
        return nil, fmt.Errorf("Redis 错误: %w", err)
    }

    // 第二步：缓存未命中，查数据库
    post, err := fetchPostFromDB(id)
    if err != nil {
        return nil, err
    }
    if post == nil {
        return nil, nil // 帖子不存在
    }

    // 第三步：回填缓存，设置过期时间
    jsonData, _ := json.Marshal(post)
    rdb.Set(ctx, key, jsonData, 10*time.Minute)

    return post, nil
}
```

### 写操作：删除缓存

```go
// 更新帖子 —— 先更新数据库，再删除缓存
func UpdatePost(post *Post) error {
    // 第一步：更新数据库（这里省略实际 DB 操作）
    log.Printf("[DB] 更新帖子 %d", post.ID)

    // 第二步：删除缓存，下次读取时会自动从 DB 加载最新值
    key := postCacheKey(post.ID)
    if err := rdb.Del(ctx, key).Err(); err != nil {
        // 删除失败不应阻断主流程，但需要记录日志
        log.Printf("[Warning] 缓存删除失败 key=%s: %v", key, err)
    }

    return nil
}
```

注意 `redis.Nil` 这个特殊错误——它表示"key 不存在"，是 go-redis 中区分"没查到"和"Redis 出错"的关键。每次 `Get` 都要处理这三种情况：命中、不存在、真正的错误。

## 3.3 TTL：缓存不能永远活着

上面代码中 `10*time.Minute` 就是 TTL（Time To Live）。没有 TTL 的缓存迟早会出问题：

- 内存无限增长，最终 Redis OOM 被杀
- 数据和数据库越来越不一致，用户看到"幽灵数据"

TTL 策略的选择取决于数据特性：

| 数据类型 | 推荐 TTL | 理由 |
|---------|---------|------|
| 帖子详情 | 5-15 分钟 | 内容相对稳定，偶尔更新 |
| 用户资料 | 10-30 分钟 | 修改频率低 |
| 排行榜快照 | 1-5 分钟 | 需要较高实时性 |
| 配置信息 | 1-24 小时 | 几乎不变 |

### 随机化 TTL：防止同时过期

一个隐蔽的坑：如果大量缓存在同一时刻设置了相同的 TTL，它们会**同时过期**，导致瞬间大量请求打到数据库——这就是**缓存雪崩**的成因之一。

```go
// 在基础 TTL 上加一个随机偏移，打散过期时间
func randomTTL(base time.Duration) time.Duration {
    // 在 base 的基础上加 0~20% 的随机偏移
    jitter := time.Duration(rand.Int63n(int64(base) / 5))
    return base + jitter
}

// 使用：
rdb.Set(ctx, key, jsonData, randomTTL(10*time.Minute))
// 实际 TTL 会在 10~12 分钟之间随机分布
```

## 3.4 缓存三大经典问题

这是面试和实战中都绑不开的三个问题。理解它们的本质比记住解决方案更重要。

### 缓存穿透：查一个根本不存在的数据

**问题**：有人反复请求 `post_id=999999`，这个帖子根本不存在。每次都缓存未命中 → 查数据库 → 返回空 → 不缓存。数据库被反复查询同一个不存在的数据。

**本质**：缓存只存了"有的东西"，对"没有的东西"毫无防御。

**解决：缓存空值**

```go
func GetPostSafe(id int64) (*Post, error) {
    key := postCacheKey(id)

    data, err := rdb.Get(ctx, key).Result()
    if err == nil {
        // 检查是否是缓存的空值标记
        if data == "NULL" {
            return nil, nil // 帖子确实不存在，直接返回
        }
        var post Post
        json.Unmarshal([]byte(data), &post)
        return &post, nil
    }
    if !errors.Is(err, redis.Nil) {
        return nil, err
    }

    // 查数据库
    post, err := fetchPostFromDB(id)
    if err != nil {
        return nil, err
    }

    if post == nil {
        // 关键：帖子不存在也缓存，但用较短的 TTL
        rdb.Set(ctx, key, "NULL", 2*time.Minute)
        return nil, nil
    }

    jsonData, _ := json.Marshal(post)
    rdb.Set(ctx, key, jsonData, randomTTL(10*time.Minute))
    return post, nil
}
```

为什么空值的 TTL 要短？因为帖子可能随后被创建，过长的空值缓存会导致用户创建了帖子却看不到。

### 缓存雪崩：大量缓存同时失效

**问题**：某一时刻大量 key 同时过期（或 Redis 宕机），所有请求瞬间涌向数据库，数据库扛不住。

**本质**：缓存层作为挡箭牌突然消失，数据库直接暴露在流量之下。

**应对**：
1. **随机 TTL**（上面已实现）：打散过期时间
2. **多级缓存**：本地缓存（如 Go 的 sync.Map 或 groupcache）+ Redis，Redis 挂了还有本地兜底
3. **熔断降级**：当数据库压力过大时，直接返回默认值或错误，不让雪崩扩散

### 缓存击穿：热点 key 过期的瞬间

**问题**：某个超级热门帖子（每秒几千次访问）的缓存恰好过期，大量请求同时发现缓存不存在，同时去查数据库。

**本质**：雪崩是大面积失效，击穿是单点失效，但效果类似。

**解决：singleflight 模式**

Go 标准扩展库提供了 `singleflight`，确保同一个 key 只有一个请求去查数据库，其他请求等待结果共享：

```go
import "golang.org/x/sync/singleflight"

var sf singleflight.Group

func GetPostWithSingleFlight(id int64) (*Post, error) {
    key := postCacheKey(id)

    // 先查缓存
    data, err := rdb.Get(ctx, key).Result()
    if err == nil {
        if data == "NULL" {
            return nil, nil
        }
        var post Post
        json.Unmarshal([]byte(data), &post)
        return &post, nil
    }
    if !errors.Is(err, redis.Nil) {
        return nil, err
    }

    // 缓存未命中：用 singleflight 保证同一 key 只查一次 DB
    result, err, _ := sf.Do(key, func() (interface{}, error) {
        post, err := fetchPostFromDB(id)
        if err != nil {
            return nil, err
        }
        if post == nil {
            rdb.Set(ctx, key, "NULL", 2*time.Minute)
            return nil, nil
        }

        jsonData, _ := json.Marshal(post)
        rdb.Set(ctx, key, jsonData, randomTTL(10*time.Minute))
        return post, nil
    })

    if err != nil {
        return nil, err
    }
    if result == nil {
        return nil, nil
    }
    return result.(*Post), nil
}
```

`singleflight` 的原理很优雅：当多个 goroutine 用同一个 key 调用 `Do` 时，只有第一个真正执行函数，其余的等待并共享返回值。这是 Go 并发原语在缓存场景的经典应用。

## 3.5 三者的关系总结

```
             什么失效了？           影响范围
穿透         不存在的数据反复查       单个 key，但可被恶意利用
雪崩         大量 key 同时过期       大面积
击穿         单个热点 key 过期       单个 key，但请求量极大

共同本质：缓存层没能挡住请求，数据库直接承压
```

记住一个总原则：**缓存的作用是保护数据库，所以缓存设计的重心不是"怎么缓存更快"，而是"缓存失效时数据库怎么不被打垮"**。

## 章末思考题

**题目一：Cache-Aside 模式中，写操作是"先更新数据库，再删除缓存"。如果反过来——"先删除缓存，再更新数据库"，会出什么问题？**

> **参考解析**：经典并发问题。请求 A 先删缓存，还没来得及更新数据库；此时请求 B 读缓存发现不存在，从数据库读到旧值并回填缓存。之后请求 A 才更新了数据库——缓存里存的是旧值，数据库是新值，不一致了。而且这个不一致会持续到缓存过期。"先更新 DB 再删缓存"虽然也有极小概率不一致（删缓存前的一小段时间内读到旧缓存），但窗口期很短且会自然修复。

**题目二：`singleflight` 解决了击穿问题，但它有一个潜在风险——如果那个唯一的数据库查询特别慢（比如 5 秒），会发生什么？你会怎么优化？**

> **参考解析**：所有等待同一个 key 的 goroutine 都会被阻塞 5 秒，如果此时请求量很大，goroutine 堆积可能耗尽内存或触发超时。优化方案：给 singleflight 包一层超时控制，用 `context.WithTimeout` 限制等待时间。或者用 `sf.DoChan` 替代 `sf.Do`，它返回 channel，可以配合 `select` 做超时：
> ```go
> ch := sf.DoChan(key, func() (interface{}, error) { ... })
> select {
> case result := <-ch:
>     return result.Val, result.Err
> case <-time.After(2 * time.Second):
>     return nil, errors.New("查询超时，请稍后重试")
> }
> ```

**题目三：我们用 `"NULL"` 字符串标记不存在的帖子来防穿透。如果攻击者用随机 ID（每次都不同）大量请求呢？缓存空值还有效吗？还有什么更好的方案？**

> **参考解析**：每个随机 ID 都会产生一个空值缓存条目，大量随机 ID 会导致 Redis 内存被垃圾空值填满，缓存空值策略在这种场景下失效。更好的方案是**布隆过滤器（Bloom Filter）**：在缓存之前加一层布隆过滤器，预先加载所有合法帖子 ID。请求进来先查布隆过滤器，如果返回"不存在"则直接拒绝，根本不查 Redis 和数据库。Redis 本身提供了 `BF.ADD`/`BF.EXISTS` 命令（需加载 RedisBloom 模块），也可以在 Go 侧用内存级布隆过滤器。代价是有极小的误判率（可能把存在的判为不存在），但在实际中完全可接受。
