# 第2章：五大数据结构与场景映射

> 上一章我们连通了 Redis 并体验了最基础的 String 读写。本章我们将看到 Redis 真正的设计精髓——**五种数据结构，每种都是对一类真实问题的直接抽象**。这正是"用正确的结构匹配正确的场景"这一核心理念的集中体现。

---

## 2.1 先建立一张映射表

在逐个学习之前，先在脑中建立这张"场景→结构"的速查表：

| 数据结构 | 你可以把它想成 | 典型场景 |
|---------|--------------|---------|
| **String** | 一个变量 | 缓存、计数器、分布式锁 |
| **Hash** | 一个 map/struct | 对象存储（用户资料、配置项） |
| **List** | 一个双端队列 | 时间线、消息队列、最近记录 |
| **Set** | 一个去重集合 | 标签、共同好友、去重统计 |
| **Sorted Set** | 一个带分数的排行榜 | 排行榜、延迟队列、范围查询 |

关键认知：**不要把 Redis 当 key-value 用完就算了**。很多人只用 String，把 JSON 塞进去——这等于放弃了 Redis 80% 的能力。选对数据结构，Redis 才能帮你在服务端直接完成计算，而不是把数据拉回 Go 里再处理。

## 2.2 Hash：用户资料存储

### 场景

我们的社区需要存用户资料。用 String 存 JSON？可以，但如果你只想更新用户的昵称，就得读取整个 JSON → 修改 → 写回。Hash 让你**直接操作单个字段**。

### 心智模型

```
Key: "user:1001"
        ┌──────────────┬──────────┐
        │  Field       │  Value   │
        ├──────────────┼──────────┤
        │  name        │  张三     │
        │  email       │  z@x.com │
        │  post_count  │  42      │
        └──────────────┴──────────┘
```

### 代码实战

```go
package main

import (
    "context"
    "fmt"
    "log"

    "github.com/redis/go-redis/v9"
)

var rdb *redis.Client
var ctx = context.Background()

func initRedis() {
    rdb = redis.NewClient(&redis.Options{Addr: "localhost:6379"})
    if err := rdb.Ping(ctx).Err(); err != nil {
        log.Fatal(err)
    }
}

// ---------- 用户资料模块 ----------

type UserProfile struct {
    Name      string `redis:"name"`
    Email     string `redis:"email"`
    PostCount int    `redis:"post_count"`
}

func userKey(id int64) string {
    return fmt.Sprintf("user:%d", id)
}

// 创建/更新用户资料
func SetUserProfile(id int64, profile *UserProfile) error {
    return rdb.HSet(ctx, userKey(id), profile).Err()
}

// 获取完整资料 —— 直接映射到结构体
func GetUserProfile(id int64) (*UserProfile, error) {
    var profile UserProfile
    err := rdb.HGetAll(ctx, userKey(id)).Scan(&profile)
    if err != nil {
        return nil, err
    }
    return &profile, nil
}

// 只更新单个字段（比如昵称），不影响其他字段
func UpdateUserName(id int64, name string) error {
    return rdb.HSet(ctx, userKey(id), "name", name).Err()
}

// 发帖计数原子递增 —— 和 String 的 INCR 类似，Hash 也有 HIncrBy
func IncrPostCount(id int64) (int64, error) {
    return rdb.HIncrBy(ctx, userKey(id), "post_count", 1).Result()
}
```

**要点**：
- `HSet` 支持直接传结构体（依靠 `redis` tag 映射字段名）
- `HGetAll` + `Scan` 反向映射回结构体，省去手动解析
- `HIncrBy` 对 Hash 内的单个数值字段做原子递增，不需要先读再写

## 2.3 List：发帖与时间线

### 场景

用户发帖后，帖子要按时间倒序出现在时间线上。List 天然就是一个有序的双端队列，左进右出（或反过来）完美匹配时间线场景。

### 心智模型

```
Key: "timeline:global"

    LPUSH 新帖子ID →  [ 1005 | 1004 | 1003 | 1002 | 1001 ]  → 旧帖子
                       ↑ 最新                          最早 ↑
    LRANGE 0 9  →  取最新 10 条
```

### 代码实战

```go
// ---------- 时间线模块 ----------

const globalTimeline = "timeline:global"

// 发布新帖 —— 将帖子 ID 推入时间线头部
func PublishPost(postID int64) error {
    // LPUSH：从左端推入，最新的帖子始终在最前面
    if err := rdb.LPush(ctx, globalTimeline, postID).Err(); err != nil {
        return err
    }
    // 保持时间线不超过 1000 条，避免无限增长
    // LTRIM：只保留索引 0~999 的元素
    return rdb.LTrim(ctx, globalTimeline, 0, 999).Err()
}

// 获取时间线 —— 分页查询
func GetTimeline(page, pageSize int) ([]string, error) {
    start := int64(page * pageSize)
    stop := start + int64(pageSize) - 1
    // LRANGE：按索引范围获取元素
    return rdb.LRange(ctx, globalTimeline, start, stop).Result()
}
```

**要点**：
- `LPUSH` + `LTRIM` 组合：推入新数据的同时裁剪长度，是 Redis 中管理"最近 N 条"数据的经典模式
- `LRANGE` 按索引取范围，天然支持分页——但注意，如果偏移量很大（比如第 10000 页），性能会下降，因为 List 是链表结构

## 2.4 Set：标签系统

### 场景

每篇帖子可以打多个标签，标签不能重复。我们还想查"某两个标签下的共同帖子"——这正是集合运算的强项。

### 心智模型

```
Key: "tag:golang"     →  { 1001, 1003, 1005 }
Key: "tag:redis"      →  { 1001, 1002, 1005 }

SINTER "tag:golang" "tag:redis"  →  { 1001, 1005 }   ← 同时有两个标签的帖子
```

### 代码实战

```go
// ---------- 标签模块 ----------

func tagKey(tag string) string {
    return fmt.Sprintf("tag:%s", tag)
}

// 为帖子添加标签
func AddTagsToPost(postID int64, tags []string) error {
    for _, tag := range tags {
        // SADD：向集合添加成员，自动去重
        if err := rdb.SAdd(ctx, tagKey(tag), postID).Err(); err != nil {
            return err
        }
    }
    return nil
}

// 查询某个标签下的所有帖子
func GetPostsByTag(tag string) ([]string, error) {
    // SMEMBERS：返回集合中的所有成员
    return rdb.SMembers(ctx, tagKey(tag)).Result()
}

// 查询同时拥有多个标签的帖子（交集）
func GetPostsByTags(tags []string) ([]string, error) {
    keys := make([]string, len(tags))
    for i, tag := range tags {
        keys[i] = tagKey(tag)
    }
    // SINTER：计算多个集合的交集
    return rdb.SInter(ctx, keys...).Result()
}

// 一个帖子有哪些标签？—— 反向索引
func postTagsKey(postID int64) string {
    return fmt.Sprintf("post:%d:tags", postID)
}

func AddTagsToPostBidirectional(postID int64, tags []string) error {
    for _, tag := range tags {
        rdb.SAdd(ctx, tagKey(tag), postID)
        rdb.SAdd(ctx, postTagsKey(postID), tag) // 反向：帖子 → 标签集合
    }
    return nil
}
```

**要点**：
- Set 的集合运算（`SINTER` 交集、`SUNION` 并集、`SDIFF` 差集）全在 Redis 服务端完成，不需要把数据拉到 Go 里再算
- 双向索引是常见模式：`tag:X → {帖子}` 和 `post:X:tags → {标签}` 同时维护

## 2.5 Sorted Set：热门排行榜

### 场景

社区需要一个"热门帖子"排行榜，按点赞数排序，支持实时更新和 Top N 查询。Sorted Set 为此而生——每个成员自带一个 score（分数），自动按分数排序。

### 心智模型

```
Key: "leaderboard:hot"

    Member     Score
    ┌─────────┬───────┐
    │ post:1005│  328  │  ← 排名第1
    │ post:1001│  215  │  ← 排名第2
    │ post:1003│  142  │  ← 排名第3
    │ post:1002│   89  │
    └─────────┴───────┘

    ZINCRBY +1  →  点赞时分数加 1，自动重排
    ZREVRANGE 0 9  →  取 Top 10
```

### 代码实战

```go
// ---------- 排行榜模块 ----------

const hotLeaderboard = "leaderboard:hot"

// 点赞 —— 分数加 1，如果帖子不在排行榜中会自动添加
func LikePost(postID int64) (float64, error) {
    member := fmt.Sprintf("post:%d", postID)
    // ZINCRBY：原子递增分数
    return rdb.ZIncrBy(ctx, hotLeaderboard, 1, member).Result()
}

// 获取 Top N 热门帖子（分数从高到低）
func GetTopPosts(n int64) ([]redis.Z, error) {
    // ZREVRANGEWITHSCORES：按分数降序取前 N 个，同时返回分数
    return rdb.ZRevRangeWithScores(ctx, hotLeaderboard, 0, n-1).Result()
}

// 查询某个帖子的排名（从 0 开始）
func GetPostRank(postID int64) (int64, error) {
    member := fmt.Sprintf("post:%d", postID)
    // ZREVRANK：降序排名，排名第1的返回 0
    return rdb.ZRevRank(ctx, hotLeaderboard, member).Result()
}
```

**要点**：
- `ZINCRBY` 是排行榜的灵魂操作：修改分数后 Redis 自动维护排序，时间复杂度 O(log N)
- `ZREVRANGE` 取降序（分数高的在前），`ZRANGE` 取升序
- Sorted Set 的强大之处：排序由 Redis 维护，你永远不需要在 Go 里 sort

## 2.6 串联起来：发帖完整流程

把本章所有模块串联到一个发帖接口中：

```go
func CreatePost(userID int64, postID int64, title string, tags []string) error {
    // 1. 用户发帖计数 +1（Hash）
    if _, err := IncrPostCount(userID); err != nil {
        return err
    }

    // 2. 推入全局时间线（List）
    if err := PublishPost(postID); err != nil {
        return err
    }

    // 3. 建立标签索引（Set）
    if err := AddTagsToPostBidirectional(postID, tags); err != nil {
        return err
    }

    // 4. 初始化排行榜分数为 0（Sorted Set）
    member := fmt.Sprintf("post:%d", postID)
    if err := rdb.ZAdd(ctx, hotLeaderboard, redis.Z{
        Score:  0,
        Member: member,
    }).Err(); err != nil {
        return err
    }

    return nil
}
```

一次发帖，四种数据结构各司其职。这就是 Redis 的设计哲学：**不做一个万能结构，而是让每种结构在自己的场景里做到极致**。

## 章末思考题

**题目一：为什么存用户资料推荐用 Hash 而不是把 JSON 字符串存到 String 里？在什么情况下，String 存 JSON 反而是更好的选择？**

> **参考解析**：Hash 的优势在于**部分读写**——可以只取或只改一个字段，省带宽也省序列化开销。但如果你的场景是"每次都读写完整对象，从不操作单个字段"，那 String 存 JSON 更简单，少一层 Hash 结构的内存开销（Hash 在字段少时用 ziplist 编码，内存很省；但字段多时会转 hashtable，反而比紧凑的 JSON 字符串占更多内存）。**选择的依据是访问模式，而非数据形状**。

**题目二：时间线用 List 实现，当数据量到百万级时会有什么问题？你能想到更好的方案吗？**

> **参考解析**：List 是链表，`LRANGE` 的时间复杂度是 O(S+N)（S 是起始偏移量）。当用户翻到很深的页码时，Redis 需要从头遍历链表，性能急剧下降。改进方案：用 **Sorted Set**，以时间戳为 score，用 `ZREVRANGEBYSCORE` 做基于游标的分页（"给我 score 小于上一页最后一条的 10 条数据"），避免偏移量问题。这也是为什么实际生产中时间线往往用 Sorted Set 而非 List。

**题目三：在 `CreatePost` 函数中，四个 Redis 操作是分别执行的。如果第3步失败了，前两步已经写入的数据怎么办？Redis 有事务能解决这个问题吗？**

> **参考解析**：Redis 的 `MULTI/EXEC` 事务能保证这几个命令**打包执行不被其他命令插入**，但不支持**回滚**——如果中间某步失败，已执行的命令不会撤销。这和 MySQL 的事务有本质区别。对于这种场景，常见策略有两个：①用 Pipeline 批量发送减少网络开销和部分失败概率；②在应用层做补偿逻辑（比如定时任务检测不一致数据）。第6章我们会深入讨论事务和 Pipeline。
