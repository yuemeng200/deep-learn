# 第5章：发布订阅与消息队列

> 前四章的所有操作都是"请求-响应"模式：客户端发命令，Redis 立刻返回结果。本章我们进入另一个维度——**异步通信**。当一个事件发生后，如何通知所有关心它的人？如何把耗时任务从请求链路中剥离出去？这依然是"用内存换时间"的体现，只不过换的不是单次查询的时间，而是**整个系统的响应时间**——把慢操作推到后台，让用户感觉一切都是即时的。

---

## 5.1 两种异步模式的区别

先厘清两个容易混淆的概念：

```
发布/订阅（Pub/Sub）              消息队列（Message Queue）
┌──────────┐                    ┌──────────┐
│ 发布者    │                    │ 生产者    │
└────┬─────┘                    └────┬─────┘
     │ 广播给所有订阅者                │ 推入队列
     ▼                               ▼
┌─────────┐                    ┌──────────┐
│ 频道     │                    │  队列     │
└─┬───┬───┘                    └────┬─────┘
  │   │   │                         │ 每条消息只被一个消费者处理
  ▼   ▼   ▼                         ▼
 A    B    C ← 所有人都收到         A（B 和 C 不会重复处理）
```

| 特性 | Pub/Sub | 消息队列 |
|------|---------|---------|
| 消费模式 | 广播（一对多） | 竞争消费（一对一） |
| 消息持久化 | 不持久化，离线就丢失 | 可以持久化在队列中 |
| 典型场景 | 实时通知、聊天室 | 异步任务、后台处理 |
| Redis 实现 | SUBSCRIBE/PUBLISH | List（LPUSH/BRPOP） |

## 5.2 Pub/Sub：实时推送新帖通知

### 场景

用户发了一篇新帖子，所有在线的"订阅者"（比如管理后台、实时 feed 页面）应该立刻收到通知。

### 心智模型

```
用户发帖 ──PUBLISH──→ 频道 "new_posts"
                          │
                ┌─────────┼─────────┐
                ▼         ▼         ▼
            管理后台   Feed服务   审核系统
            （订阅者） （订阅者） （订阅者）
```

### 发布端

```go
package main

import (
    "context"
    "encoding/json"
    "fmt"
    "log"

    "github.com/redis/go-redis/v9"
)

var rdb *redis.Client
var ctx = context.Background()

func initRedis() {
    rdb = redis.NewClient(&redis.Options{Addr: "localhost:6379"})
    if err := rdb.Ping(ctx).Err(); err != nil {
        log.Fatal(err)
    }
}

// ---------- Pub/Sub 模块 ----------

const channelNewPosts = "new_posts"

type PostEvent struct {
    PostID int64  `json:"post_id"`
    Title  string `json:"title"`
    Author string `json:"author"`
    Action string `json:"action"` // "created", "updated", "deleted"
}

// 发布新帖事件
func PublishPostEvent(event *PostEvent) error {
    data, _ := json.Marshal(event)
    // PUBLISH：向频道发送消息，返回收到消息的订阅者数量
    receivers, err := rdb.Publish(ctx, channelNewPosts, data).Result()
    if err != nil {
        return fmt.Errorf("发布事件失败: %w", err)
    }
    log.Printf("[Pub/Sub] 事件已发布，%d 个订阅者收到", receivers)
    return nil
}
```

### 订阅端

```go
// 订阅新帖事件 —— 通常在一个独立的 goroutine 中运行
func SubscribePostEvents(ctx context.Context, handler func(*PostEvent)) error {
    sub := rdb.Subscribe(ctx, channelNewPosts)

    // 确认订阅成功
    _, err := sub.Receive(ctx)
    if err != nil {
        return fmt.Errorf("订阅失败: %w", err)
    }
    log.Printf("[Pub/Sub] 已订阅频道: %s", channelNewPosts)

    // 获取消息通道
    ch := sub.Channel()

    for {
        select {
        case msg, ok := <-ch:
            if !ok {
                log.Println("[Pub/Sub] 频道已关闭")
                return nil
            }
            var event PostEvent
            if err := json.Unmarshal([]byte(msg.Payload), &event); err != nil {
                log.Printf("[Pub/Sub] 消息解析失败: %v", err)
                continue
            }
            handler(&event)

        case <-ctx.Done():
            sub.Close()
            return ctx.Err()
        }
    }
}
```

### 在服务中使用

```go
func main() {
    initRedis()

    // 启动订阅者（后台 goroutine）
    ctx, cancel := context.WithCancel(context.Background())
    defer cancel()

    go func() {
        err := SubscribePostEvents(ctx, func(event *PostEvent) {
            log.Printf("[通知] 新帖子！ID=%d 标题=%s 作者=%s",
                event.PostID, event.Title, event.Author)
            // 这里可以：推送 WebSocket、更新缓存、发送邮件...
        })
        if err != nil {
            log.Printf("订阅异常退出: %v", err)
        }
    }()

    // 模拟发帖
    PublishPostEvent(&PostEvent{
        PostID: 2001,
        Title:  "Redis Pub/Sub 实战",
        Author: "张三",
        Action: "created",
    })

    // 保持主程序运行...
    select {}
}
```

### Pub/Sub 的关键局限

必须正视 Pub/Sub 的局限性，否则在生产中会踩坑：

1. **Fire and forget**：消息不持久化。如果订阅者掉线，期间发布的消息全部丢失
2. **没有 ACK 机制**：无法确认订阅者是否成功处理了消息
3. **没有消息积压**：发布者不关心有没有订阅者，消息发了就算完成

这意味着 Pub/Sub 只适合**"丢了也没关系"**的场景：实时通知、缓存失效广播、开发调试。如果消息不能丢，需要用消息队列。

## 5.3 List 消息队列：可靠的异步任务

### 场景

用户发帖后，需要做几件耗时的事：生成摘要、检测违规内容、发送通知邮件。这些不应该阻塞发帖接口的响应，应该推到后台异步处理。

### 心智模型

```
发帖接口（生产者）
    │
    │ LPUSH 任务
    ▼
┌───────────────────────────┐
│  队列 "queue:post_tasks"   │
│  [ task3 | task2 | task1 ] │
└───────────────┬───────────┘
                │ BRPOP 取出任务（阻塞等待）
                ▼
         后台 Worker（消费者）
         处理任务...
```

### 实现

```go
// ---------- 消息队列模块 ----------

const taskQueue = "queue:post_tasks"

type Task struct {
    ID        string `json:"id"`
    Type      string `json:"type"` // "generate_summary", "check_content", "send_notification"
    PostID    int64  `json:"post_id"`
    CreatedAt int64  `json:"created_at"`
}

// 生产者：推送任务到队列
func EnqueueTask(ctx context.Context, task *Task) error {
    data, _ := json.Marshal(task)
    // LPUSH：从左端推入，新任务排在队尾（配合 BRPOP 从右端取，实现 FIFO）
    return rdb.LPush(ctx, taskQueue, data).Err()
}

// 消费者：从队列中取任务并处理
func StartWorker(ctx context.Context, handler func(*Task) error) {
    log.Println("[Worker] 启动，等待任务...")

    for {
        select {
        case <-ctx.Done():
            log.Println("[Worker] 收到停止信号，退出")
            return
        default:
        }

        // BRPOP：阻塞式从右端弹出
        // 如果队列为空，会阻塞等待，不浪费 CPU（超时 5 秒后重试）
        result, err := rdb.BRPop(ctx, 5*time.Second, taskQueue).Result()
        if err != nil {
            if errors.Is(err, redis.Nil) {
                continue // 超时，重新等待
            }
            if ctx.Err() != nil {
                return // context 取消
            }
            log.Printf("[Worker] 取任务出错: %v", err)
            time.Sleep(time.Second) // 避免错误时疯狂重试
            continue
        }

        // result[0] 是 key 名，result[1] 是值
        var task Task
        if err := json.Unmarshal([]byte(result[1]), &task); err != nil {
            log.Printf("[Worker] 任务解析失败: %v", err)
            continue
        }

        log.Printf("[Worker] 处理任务: type=%s post_id=%d", task.Type, task.PostID)
        if err := handler(&task); err != nil {
            log.Printf("[Worker] 任务处理失败: %v", err)
            // 失败的任务可以重新入队（简易重试）
            rdb.LPush(ctx, taskQueue, result[1])
        }
    }
}
```

**`BRPOP` 是核心**——B 代表 Blocking。和 `RPOP` 的区别：

| 命令 | 队列为空时 | CPU 消耗 |
|------|----------|---------|
| `RPOP`（轮询） | 立即返回 nil，需要 sleep 后重试 | 高（忙等待） |
| `BRPOP`（阻塞） | 挂起等待，有数据时立即唤醒 | 几乎为零 |

### 发帖时投递异步任务

```go
// 在发帖流程中加入异步任务投递
func CreatePostWithAsync(userID, postID int64, title string, tags []string) error {
    // ... 前面的同步操作（写时间线、标签、排行榜）保持不变 ...

    // 投递异步任务 —— 不阻塞主流程
    tasks := []Task{
        {ID: fmt.Sprintf("task:%d:summary", postID), Type: "generate_summary", PostID: postID, CreatedAt: time.Now().Unix()},
        {ID: fmt.Sprintf("task:%d:check", postID), Type: "check_content", PostID: postID, CreatedAt: time.Now().Unix()},
        {ID: fmt.Sprintf("task:%d:notify", postID), Type: "send_notification", PostID: postID, CreatedAt: time.Now().Unix()},
    }

    for _, t := range tasks {
        task := t
        if err := EnqueueTask(ctx, &task); err != nil {
            log.Printf("[Warning] 任务投递失败: %v", err) // 记日志但不影响主流程
        }
    }

    // 同时发布 Pub/Sub 事件（给在线的实时订阅者）
    PublishPostEvent(&PostEvent{PostID: postID, Title: title, Action: "created"})

    return nil
}
```

### 启动多个 Worker

```go
func main() {
    initRedis()

    ctx, cancel := context.WithCancel(context.Background())
    defer cancel()

    // 启动 3 个 Worker 并发消费（Redis 的 BRPOP 保证每条消息只被一个 Worker 取到）
    for i := 0; i < 3; i++ {
        workerID := i
        go func() {
            StartWorker(ctx, func(task *Task) error {
                switch task.Type {
                case "generate_summary":
                    log.Printf("[Worker %d] 生成摘要: post %d", workerID, task.PostID)
                    time.Sleep(200 * time.Millisecond) // 模拟耗时
                case "check_content":
                    log.Printf("[Worker %d] 内容审核: post %d", workerID, task.PostID)
                    time.Sleep(500 * time.Millisecond)
                case "send_notification":
                    log.Printf("[Worker %d] 发送通知: post %d", workerID, task.PostID)
                    time.Sleep(100 * time.Millisecond)
                }
                return nil
            })
        }()
    }

    // 启动 HTTP 服务...
}
```

多个 Worker 同时 `BRPOP` 同一个队列时，Redis 保证每条消息**只会被一个 Worker 取到**——这是天然的竞争消费模型，不需要额外加锁。

## 5.4 List 队列的局限与 Stream 简介

List 队列简单好用，但有两个硬伤：

**问题一：没有 ACK 机制**

`BRPOP` 取出消息后，消息就从队列中消失了。如果 Worker 取到消息后崩溃了，消息就丢了。

一个简易的解决方案是用 `BRPOPLPUSH`（原子地从一个队列弹出并推入另一个队列）：

```go
// 可靠消费模式：取出任务的同时放入"处理中"队列
func ReliableDequeue(ctx context.Context) (string, error) {
    // 从 taskQueue 弹出，同时推入 processingQueue
    processingQueue := taskQueue + ":processing"
    return rdb.BRPopLPush(ctx, taskQueue, processingQueue, 5*time.Second).Result()
}

// 处理成功后，从"处理中"队列移除
func AckTask(ctx context.Context, taskData string) error {
    processingQueue := taskQueue + ":processing"
    return rdb.LRem(ctx, processingQueue, 1, taskData).Err()
}

// 定时任务：将"处理中"队列里超时的任务重新入队
func RecoverStuckTasks(ctx context.Context) {
    processingQueue := taskQueue + ":processing"
    tasks, _ := rdb.LRange(ctx, processingQueue, 0, -1).Result()
    for _, t := range tasks {
        var task Task
        json.Unmarshal([]byte(t), &task)
        // 如果任务在处理中队列里超过 5 分钟，认为 Worker 崩溃了
        if time.Now().Unix()-task.CreatedAt > 300 {
            rdb.LRem(ctx, processingQueue, 1, t)
            rdb.LPush(ctx, taskQueue, t) // 重新入队
            log.Printf("[Recovery] 任务重新入队: %s", task.ID)
        }
    }
}
```

**问题二：没有消费者组**

如果你有多个不同的消费者组（比如"审核组"和"通知组"）都需要处理同一条消息，List 做不到——消息被一个 BRPOP 取走后就没了。

**Redis Stream**（5.0+）解决了这两个问题：内置 ACK、消费者组、消息持久化。如果你的场景需要可靠消息处理，Stream 是更好的选择。但 Stream 的 API 更复杂，本课程聚焦于核心概念，不做展开——掌握了 List 队列的思想后，迁移到 Stream 只是 API 的学习成本。

## 5.5 Pub/Sub vs List 队列：如何选择

```
                    Pub/Sub                    List 队列
需要广播？           ✓ 一对多                   ✗ 一对一
消息能丢？           ✓ 可接受                   ✗ 不行
需要离线消费？       ✗ 不支持                   ✓ 消息持久在队列
需要确认处理？       ✗ 没有 ACK                 △ 需自己实现

选择原则：
  "通知类"（实时、可丢） → Pub/Sub
  "任务类"（可靠、有序） → List 队列（或 Stream）
  两者都要             → 同时用！发帖时既 PUBLISH 又 LPUSH
```

## 章末思考题

**题目一：Pub/Sub 消息不持久化，订阅者掉线就会丢消息。如果你要实现一个"不丢消息的广播系统"（多个消费者组都能收到每条消息），用 Redis 怎么实现？**

> **参考解析**：两种方案。方案一：为每个消费者组维护一个独立的 List 队列，生产者发布时向所有队列都 LPUSH 一份——简单但不够优雅，新增消费者组需要改生产者代码。方案二：使用 **Redis Stream** + **消费者组（Consumer Group）**。Stream 天生支持多个消费者组独立消费同一条消息流，每个组有自己的偏移量和 ACK 状态。这正是 Stream 被设计出来要解决的问题。

**题目二：在 `StartWorker` 中，任务处理失败后我们简单地重新 LPUSH 回队列。这个"重试"策略有什么隐患？你会怎么改进？**

> **参考解析**：如果某条消息本身有问题（比如数据格式错误），每次处理都会失败，然后被重新入队，形成**毒消息（poison message）**——它会无限循环消费，占用 Worker 资源并产生大量错误日志。改进：在 Task 结构中加一个 `retry_count` 字段，每次重试时加 1。超过最大重试次数（比如 3 次）后，不再入队而是推入一个**死信队列（Dead Letter Queue）**，由人工或专门的程序后续处理。这是消息队列设计中的标准模式。

**题目三：我们启动了 3 个 Worker goroutine 消费同一个队列。如果某类任务（比如内容审核）特别慢，会影响其他任务（比如发送通知）的处理速度吗？你会怎么设计来避免？**

> **参考解析**：会。3 个 Worker 是通用的，如果都在处理慢任务，快任务就得排队等。这叫**队头阻塞（Head-of-Line Blocking）**。解决方案：**按任务类型拆分队列**。`queue:summary`、`queue:check`、`queue:notify` 各自独立，每种队列配置不同数量的 Worker。慢任务队列可以多分配 Worker，快任务队列少分配。这样慢任务不会拖累快任务，也方便单独监控每种任务的积压情况。
