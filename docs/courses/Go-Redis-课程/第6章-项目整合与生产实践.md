## 6.2 连接池调优

第1章我们用了默认配置。生产环境需要根据实际负载调整：

```go
package redis

import (
    "context"
    "fmt"
    "log"
    "time"

    "github.com/redis/go-redis/v9"
)

type Config struct {
    Addr         string
    Password     string
    DB           int
    PoolSize     int
    MinIdleConns int
    DialTimeout  time.Duration
    ReadTimeout  time.Duration
    WriteTimeout time.Duration
}

func NewClient(cfg *Config) *redis.Client {
    rdb := redis.NewClient(&redis.Options{
        Addr:     cfg.Addr,
        Password: cfg.Password,
        DB:       cfg.DB,

        // ---------- 连接池 ----------
        PoolSize:     cfg.PoolSize,     // 最大连接数
        MinIdleConns: cfg.MinIdleConns, // 最小空闲连接（预热，避免冷启动延迟）

        // ---------- 超时 ----------
        DialTimeout:  cfg.DialTimeout,  // 建立连接超时
        ReadTimeout:  cfg.ReadTimeout,  // 单次读超时
        WriteTimeout: cfg.WriteTimeout, // 单次写超时
    })

    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()

    if err := rdb.Ping(ctx).Err(); err != nil {
        log.Fatalf("Redis 连接失败: %v", err)
    }

    return rdb
}
```

### 连接池大小怎么定？

一个经验公式：

```
PoolSize = 预期并发请求数 × 每个请求的 Redis 调用次数 × 平均调用耗时 / 1秒
```

举例：并发 200 请求，每个请求平均 3 次 Redis 调用，每次耗时约 1ms：

```
PoolSize = 200 × 3 × 0.001 / 1 ≈ 0.6 → 实际设 10~20 就绑绑有余
```

Redis 调用极快（通常 <1ms），所以连接池不需要很大。**设太大反而有害**：每个连接占用 Redis 服务端的内存和文件描述符，连接多了 Redis 反而变慢。

### 监控连接池状态

```go
// 定期打印连接池状态，帮助判断配置是否合理
func MonitorPool(rdb *redis.Client, interval time.Duration) {
    ticker := time.NewTicker(interval)
    for range ticker.C {
        stats := rdb.PoolStats()
        log.Printf("[Pool] 总连接=%d 空闲=%d 过期=%d 命中=%d 未命中=%d 超时=%d",
            stats.TotalConns,
            stats.IdleConns,
            stats.StaleConns,
            stats.Hits,     // 从池中成功获取连接的次数
            stats.Misses,   // 池中无可用连接，需要新建的次数
            stats.Timeouts, // 等待连接超时的次数（这个数值应该为0！）
        )
    }
}
```

**关键指标**：如果 `Timeouts > 0`，说明连接池太小或有连接泄漏；如果 `Misses` 持续很高，说明 `MinIdleConns` 设得太低。

## 6.3 Pipeline：批量操作的利器

前面章节中，发帖操作涉及 4-5 个 Redis 命令，每个命令都是一次网络往返：

```
Go ──SET──→ Redis ──响应──→ Go     ← 1 RTT
Go ──LPUSH─→ Redis ──响应──→ Go    ← 1 RTT
Go ──SADD──→ Redis ──响应──→ Go    ← 1 RTT
Go ──ZADD──→ Redis ──响应──→ Go    ← 1 RTT
                                    共 4 RTT
```

如果网络延迟 0.5ms，4 次往返就是 2ms。看似不多，但在高并发下会累积。Pipeline 把多个命令打包成一次发送：

```
Go ──[SET, LPUSH, SADD, ZADD]──→ Redis ──[4个响应]──→ Go
                                                        共 1 RTT
```

### 用 Pipeline 重写发帖流程

```go
func CreatePostPipelined(ctx context.Context, userID, postID int64, tags []string) error {
    pipe := rdb.Pipeline()

    // 1. 用户发帖计数 +1
    userKey := fmt.Sprintf("user:%d", userID)
    pipe.HIncrBy(ctx, userKey, "post_count", 1)

    // 2. 推入全局时间线
    pipe.LPush(ctx, "timeline:global", postID)
    pipe.LTrim(ctx, "timeline:global", 0, 999)

    // 3. 标签索引
    postMember := fmt.Sprintf("post:%d", postID)
    for _, tag := range tags {
        pipe.SAdd(ctx, fmt.Sprintf("tag:%s", tag), postID)
        pipe.SAdd(ctx, fmt.Sprintf("post:%d:tags", postID), tag)
    }

    // 4. 排行榜初始化
    pipe.ZAdd(ctx, "leaderboard:hot", redis.Z{Score: 0, Member: postMember})

    // 一次性发送所有命令
    cmds, err := pipe.Exec(ctx)
    if err != nil {
        return fmt.Errorf("Pipeline 执行失败: %w", err)
    }

    // 检查每个命令是否成功（Pipeline 不会因为单个命令失败而中断其他命令）
    for i, cmd := range cmds {
        if cmd.Err() != nil {
            log.Printf("[Pipeline] 第 %d 个命令失败: %v", i, cmd.Err())
        }
    }

    return nil
}
```

**Pipeline vs MULTI/EXEC 的区别**：

| 特性 | Pipeline | MULTI/EXEC（事务） |
|------|---------|-------------------|
| 减少网络往返 | 是 | 是 |
| 原子执行 | 否（命令之间可能插入其他客户端的命令） | 是（所有命令要么全执行，要么全不执行） |
| 单个命令失败 | 不影响其他命令 | 不影响其他命令（但注意：不会回滚！） |
| 性能 | 更快 | 略慢（有 MULTI/EXEC 开销） |

大多数场景用 Pipeline 就够了。只有当你需要"这些命令之间不被其他客户端打断"时，才需要事务。

## 6.4 Lua 脚本：当原子性真正重要时

Pipeline 和 MULTI/EXEC 都有局限：无法在命令之间做**条件判断**。比如"如果库存大于 0，减 1 并返回成功"——读和写之间不能有间隙。

Lua 脚本在 Redis 服务端原子执行，完美解决这个问题：

```go
// 示例：原子的"检查并扣减"操作
// 用于限流场景更精确的版本，或者库存扣减
var decrIfPositiveScript = redis.NewScript(`
    local current = tonumber(redis.call('GET', KEYS[1]) or '0')
    if current > 0 then
        redis.call('DECR', KEYS[1])
        return 1  -- 成功
    end
    return 0  -- 不足
`)

func DecrIfPositive(ctx context.Context, key string) (bool, error) {
    result, err := decrIfPositiveScript.Run(ctx, rdb, []string{key}).Int()
    if err != nil {
        return false, err
    }
    return result == 1, nil
}
```

再来一个实战的例子——用 Lua 实现精确的滑动窗口限流（第4章 Pipeline 版的原子升级）：

```go
var slidingWindowScript = redis.NewScript(`
    local key = KEYS[1]
    local now = tonumber(ARGV[1])
    local window = tonumber(ARGV[2])
    local limit = tonumber(ARGV[3])

    -- 清除窗口外的旧记录
    redis.call('ZREMRANGEBYSCORE', key, 0, now - window)

    -- 统计当前窗口内的请求数
    local count = redis.call('ZCARD', key)

    if count < limit then
        -- 未超限，添加当前请求
        redis.call('ZADD', key, now, now .. ':' .. math.random(1000000))
        redis.call('EXPIRE', key, math.ceil(window / 1000000))
        return 1  -- 放行
    end

    return 0  -- 限流
`)

func AtomicRateLimit(ctx context.Context, userID string, limit int64, window time.Duration) (bool, error) {
    key := fmt.Sprintf("ratelimit:%s", userID)
    now := time.Now().UnixMicro()

    result, err := slidingWindowScript.Run(ctx, rdb,
        []string{key},
        now,
        window.Microseconds(),
        limit,
    ).Int()

    if err != nil {
        return false, err
    }
    return result == 1, nil
}
```

**Lua 脚本的注意事项**：
- 脚本在 Redis 中**单线程原子执行**，执行期间其他所有命令都被阻塞
- 所以脚本必须短小精悍，绝不能做耗时操作（比如循环百万次）
- `redis.NewScript` 会自动缓存脚本的 SHA1，第二次调用只传 SHA1 而不是完整脚本，节省带宽

## 6.5 错误处理与降级策略

生产环境中 Redis 不是永远可用的。网络抖动、主从切换、内存满了都可能导致短暂故障。关键原则：**Redis 挂了，你的服务不应该跟着挂**。

```go
// 带降级的缓存读取
func GetPostWithFallback(ctx context.Context, id int64) (*Post, error) {
    key := postCacheKey(id)

    // 尝试读缓存
    data, err := rdb.Get(ctx, key).Result()
    if err == nil {
        var post Post
        json.Unmarshal([]byte(data), &post)
        return &post, nil
    }

    if !errors.Is(err, redis.Nil) {
        // Redis 出错了（不是"key不存在"，而是连接问题等）
        log.Printf("[Cache] Redis 错误，降级到数据库: %v", err)
        // 不 return error —— 降级到数据库
    }

    // 从数据库获取
    post, err := fetchPostFromDB(id)
    if err != nil {
        return nil, err
    }
    if post == nil {
        return nil, nil
    }

    // 尝试回填缓存（失败也不影响主流程）
    if jsonData, err := json.Marshal(post); err == nil {
        if err := rdb.Set(ctx, key, jsonData, randomTTL(10*time.Minute)).Err(); err != nil {
            log.Printf("[Cache] 回填缓存失败（已降级）: %v", err)
        }
    }

    return post, nil
}
```

### 分层降级策略

```
          正常模式                    Redis 故障模式
         ┌─────────┐                ┌─────────┐
请求 ──→ │ 限流检查  │           ──→ │ 跳过限流  │  ← 放行（宁可放过）
         └────┬────┘                └────┬────┘
              ▼                          ▼
         ┌─────────┐                ┌─────────┐
         │ 读缓存   │           ──→ │ 直查数据库│  ← 降级
         └────┬────┘                └────┬────┘
              ▼                          ▼
         ┌─────────┐                ┌─────────┐
         │ 查会话   │           ──→ │ JWT 兜底 │  ← 备用方案（如果有的话）
         └─────────┘                └─────────┘
```

## 6.6 优雅关闭

服务停止时需要有序清理资源，避免丢数据或连接泄漏：

```go
func main() {
    rdb := initRedis()

    // 创建可取消的 context
    ctx, cancel := context.WithCancel(context.Background())

    // 启动后台 Worker
    var wg sync.WaitGroup
    for i := 0; i < 3; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            StartWorker(ctx, taskHandler)
        }()
    }

    // 启动 Pub/Sub 订阅
    wg.Add(1)
    go func() {
        defer wg.Done()
        SubscribePostEvents(ctx, eventHandler)
    }()

    // 启动 HTTP 服务
    srv := &http.Server{Addr: ":8080"}
    go func() {
        if err := srv.ListenAndServe(); err != http.ErrServerClosed {
            log.Fatalf("HTTP 服务异常: %v", err)
        }
    }()
    log.Println("服务启动于 :8080")

    // 等待中断信号
    sigCh := make(chan os.Signal, 1)
    signal.Notify(sigCh, syscall.SIGINT, syscall.SIGTERM)
    <-sigCh
    log.Println("收到停止信号，开始优雅关闭...")

    // 1. 停止接收新请求
    shutdownCtx, shutdownCancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer shutdownCancel()
    srv.Shutdown(shutdownCtx)

    // 2. 停止后台任务
    cancel() // 取消 context，Worker 和 Subscriber 会自行退出

    // 3. 等待后台任务完成
    wg.Wait()

    // 4. 关闭 Redis 连接池
    rdb.Close()

    log.Println("服务已停止")
}
```

关闭顺序很重要：**先停入口（HTTP）→ 再停处理（Worker）→ 最后关连接（Redis）**。反过来会导致正在处理的请求因为 Redis 连接已关闭而报错。

## 6.7 完整项目回顾

回看我们的社区后端，一个请求的完整生命周期：

```
客户端 POST /post/create
    │
    ▼
[限流中间件] ── Redis ZRANGEBYSCORE（滑动窗口）── 超限则 429
    │
    ▼
[会话中间件] ── Redis GET sess:xxx ── 无效则 401
    │
    ▼
[发帖Handler]
    │
    ├── Redis HIncrBy user:1001 post_count   ← Hash
    ├── Redis LPUSH timeline:global          ← List
    ├── Redis SADD tag:golang                ← Set
    ├── Redis ZADD leaderboard:hot           ← Sorted Set
    │   （以上通过 Pipeline 批量执行，1 RTT）
    │
    ├── Redis LPUSH queue:post_tasks         ← 异步任务入队
    ├── Redis PUBLISH new_posts              ← 实时通知
    │
    └── 返回 201 Created

后台：
    Worker ── BRPOP queue:post_tasks ── 处理异步任务
    Subscriber ── 接收 Pub/Sub ── 推送实时通知
```

五种数据结构、缓存、会话、限流、消息队列——全部在一个真实的请求流程中串联起来。

## 章末思考题

**题目一：Pipeline 将多个命令打包发送以减少 RTT。但如果打包了 1000 个命令，会有什么问题？Pipeline 的批次大小有最佳实践吗？**

> **参考解析**：两个问题。①**内存**：Pipeline 会在客户端缓冲所有命令，在服务端缓冲所有响应。1000 个命令如果每个响应 1KB，就是 1MB 的内存——在高并发下会累积。②**阻塞**：Pipeline 执行期间虽然不像 MULTI/EXEC 那样有原子性保证，但大量命令打包会让 Redis 在短时间内忙于处理这一批，增加其他客户端的延迟。最佳实践：每批 **50~200 个命令**，超过的做分批发送。go-redis 也提供了 `Pipeliner` 接口，可以方便地做分批。

**题目二：我们的优雅关闭中，Worker 收到 cancel 信号后会退出。但如果 Worker 正在处理一个任务（比如内容审核执行到一半），会发生什么？怎么保证任务不丢？**

> **参考解析**：`ctx.Done()` 只是通知 Worker "该停了"，但 Worker 不会立刻中断正在执行的任务——它会在当前任务处理完后、下一次 `select` 检查时才退出。这是合理的。真正的风险是：如果任务特别慢（比如 30 秒），`wg.Wait()` 会等很久。可以给 Wait 加超时：超时后强制退出，未完成的任务会因为 BRPOP 已经取出而丢失。解决方案：使用前面讲过的 `BRPOPLPUSH` 可靠消费模式，未 ACK 的任务留在 processing 队列中，下次启动时由 `RecoverStuckTasks` 重新入队。

**题目三：整个项目中，Redis 同时承担了缓存、会话、限流、消息队列多个角色。这样做有什么风险？在更大规模的系统中，你会怎么调整架构？**

> **参考解析**：**单点压力集中**。如果某个角色的负载暴增（比如限流产生大量写入），会影响其他角色（比如缓存读取变慢）。而且不同角色对 Redis 的要求不同：缓存丢了可以重建，但会话丢了用户要重新登录。改进方案：**按角色拆分 Redis 实例**。至少分两组：一组给"可丢失"的数据（缓存、限流计数），配置 `maxmemory-policy allkeys-lru` 允许自动淘汰；一组给"不可丢失"的数据（会话、消息队列），开启 AOF 持久化。消息队列在更大规模下应该迁移到专业的消息中间件（Kafka、RabbitMQ），让 Redis 专注于它最擅长的事——快速读写。
