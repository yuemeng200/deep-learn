# 结业总结

## ① 核心理念提炼

**理念一：日志排查的本质是"提问能力"**
工具只是手段，真正决定排查效率的是你**问对了什么问题**。"影响了多大范围？什么时候开始的？错误类型是什么？流量有没有异常？"——每一个正确的问题都让你离答案更近一步。grep 和 awk 谁都会敲，但知道此刻该 grep 什么才是核心竞争力。

**理念二：分层思维——从应用到内核逐层下探**
应用日志看不到的答案，去系统日志找；系统日志看不到的，去内核日志找。OOM、磁盘故障、网络中断这类问题，在应用层是"隐形"的。养成"从上往下查"的习惯，不要只盯着自己的代码。

**理念三：写日志和查日志是同一件事的两面**
你排查时的痛苦，根源往往在于当初日志写得不好。结构化、带标识、一事一行——这些不是教条，而是"未来的你"能不能在凌晨 3 点快速定位问题的决定因素。

**理念四：管道组合是 Linux 的灵魂**
不需要记住每个命令的所有参数。记住每个工具"做什么"（grep 筛、awk 提、sed 换、sort 排、uniq 数、jq 解 JSON），然后用管道像搭积木一样组合。逐步构建、中间 head 检查——这个方法论比任何单个命令都重要。

**理念五：先止血，再治本**
线上排查永远是"先恢复服务，再找根因"。临时增大连接数、临时重启、临时降级都可以——但止血之后必须跟进根本修复。很多线上事故的复发，就是因为止血后就忘了。

## ② 学员评估与定位

- **起点**：开始时"偶尔用过"Linux 命令，对日志没有系统认知。
- **接收效率**：全程连续推进 7 章密集内容，信息吸收能力不错。部分细节（awk 条件语法、jq select 表达式）可能还停留在"看懂了"但没"练熟"。
- **当前定位**：从**入门**提升到**初级偏上**。已建立完整知识框架，要到"中级"需要 3-5 次真实问题的实战练习。
- **建议**：在自己的 CentOS 上复现第7章实战场景，写一个会故意报错的 Go 服务，只用命令行完成排查。读十遍不如练一遍。

## ③ 进阶方向

**方向一：分布式链路追踪（OpenTelemetry）**
从"grep 单机日志"升级到"在分布式系统中追踪任意请求的完整生命周期"。学习 trace_id 传递、跨服务链路串联、Jaeger 可视化。

**方向二：日志采集与分析平台（ELK / Loki + Grafana）**
当日志量大到单机 grep 搞不定时，集中式日志平台可以在 Web 界面做复杂查询、看趋势图、设置告警。

**方向三：Linux 性能诊断（top/htop/iostat/strace）**
从"看日志"到"看系统运行时状态"。top 看 CPU、free 看内存、iostat 看磁盘 IO、strace 跟踪系统调用、pprof 分析 Go 程序热点。

**方向四：自动化运维脚本与告警**
把排查经验沉淀为脚本：自动收集现场、自动分析常见故障模式、自动发通知。从"人肉排查"到"脚本辅助"再到"半自动化诊断"。
