# 第6章：管道组合术 — 复杂场景的排查链路

> 前面几章你学了一把把独立的"刀"——grep 筛选、awk 提取、sed 清洗、sort/uniq 统计、jq 解析 JSON。本章是**把刀法串成招式**。管道 `|` 是 Linux 的灵魂：每个工具只做一件事，通过管道组合成无限可能。这正是"一切皆文件、小工具组合"这一 Unix 设计哲学的终极体现。

---

## 6.1 管道的本质

管道 `|` 做的事很简单：**把前一个命令的 stdout（标准输出）接到后一个命令的 stdin（标准输入）**。

```
命令A 的输出  →  管道  →  命令B 的输入  →  管道  →  命令C 的输入
```

类比：工厂流水线。原材料（日志文件）进入第一道工序（grep 筛选），产出半成品传给第二道工序（awk 提取），再传给第三道（sort 排序）……最终产出成品（你要的答案）。

---

## 6.2 构建思路：从问题到命令链

面对一个排查需求，不要试图一口气写出完整命令。**逐步构建**：

**需求**：找出最近 1 小时内响应最慢的 10 个请求，包含 uid 和接口路径。

```bash
# 第1步：筛选时间
grep "2024-03-15 10:" app.log

# 第2步：只要有延迟的行
grep "2024-03-15 10:" app.log | grep "latency"

# 第3步：提取关键字段
grep "2024-03-15 10:" app.log | grep "latency" | awk '{print $2, $6, $7, $8}'

# 第4步：清洗 + 排序 + 取 top
grep "2024-03-15 10:" app.log | grep "latency" | sed 's/latency=//;s/ms / /' | awk '{print $8, $6, $7}' | sort -rn | head -n 10
```

**关键心法**：每加一个管道就 `| head` 看看中间结果，确认正确后再往后加。

## 6.3 结构化日志（JSON）下的管道组合

```bash
# 最近 1 小时响应最慢的 10 个请求
journalctl -u order-service --since "1 hour ago" -o cat --no-pager \
  | jq -r 'select(.latency_ms != null) | [.latency_ms, .uid, .path, .time] | @tsv' \
  | sort -rn \
  | head -n 10
```

拆解：
1. `journalctl ... -o cat` — 取原始 JSON 日志
2. `jq -r 'select(...) | [...] | @tsv'` — 筛选、提取、输出为 Tab 分隔
3. `sort -rn` — 按延迟倒序
4. `head -n 10` — 取前 10

## 6.4 xargs — 把输出变成参数

管道传递"文本流"，但有时需要把输出**作为另一个命令的参数**。

```bash
# 找到包含 "panic" 的日志文件，用 less 逐个查看
grep -rl "panic" /var/log/ | xargs less

# 找到旧日志文件，批量删除
find /var/log/myapp/ -name "*.log" -mtime +7 | xargs rm -v

# 占位符模式：-I {}
grep "ERROR" app.log | awk -F'"uid":' '{print $2}' | awk -F',' '{print $1}' | sort -u | xargs -I {} echo "SELECT * FROM users WHERE id = {};"
```

### xargs 安全参数

```bash
-p                    # 执行前确认
-t                    # 打印要执行的命令
--no-run-if-empty     # 输入为空就不执行
```

## 6.5 tee — 分流：既看又存

`tee` 像 T 型水管——数据同时流向屏幕和文件。

```bash
grep "ERROR" app.log | tee errors.txt | wc -l
```

**排查场景：保存中间结果**

```bash
journalctl -u order-service --since "1 hour ago" -o cat --no-pager \
  | tee /tmp/raw-logs.json \
  | jq 'select(.level=="ERROR")' \
  | tee /tmp/errors.json \
  | jq -r '.err' \
  | sort | uniq -c | sort -rn
```

追加写入：`tee -a file`

## 6.6 进程替换 — 同时比对两组数据

```bash
# 对比今天和昨天的错误类型差异
diff <(grep "ERROR" today.log | awk '{print $6}' | sort -u) \
     <(grep "ERROR" yesterday.log | awk '{print $6}' | sort -u)

# 对比两个时段的错误
diff <(journalctl -u order-service --since "09:00" --until "10:00" -o cat --no-pager | jq -r '.err' | sort -u) \
     <(journalctl -u order-service --since "10:00" --until "11:00" -o cat --no-pager | jq -r '.err' | sort -u)
```

`<(...)` 把命令输出伪装成"文件"交给 diff。

## 6.7 实战：六个真实排查场景

### 场景一：错误率突增，快速定位是哪个接口

```bash
journalctl -u order-service --since "10 min ago" -o cat --no-pager \
  | jq -r 'select(.level=="ERROR") | .path' \
  | sort | uniq -c | sort -rn
```

### 场景二：追踪某个用户的全部请求

```bash
journalctl -u order-service --since today -o cat --no-pager \
  | jq 'select(.uid=="1004")' \
  | jq -r '[.time, .level, .msg, .err // ""] | @tsv'
```

### 场景三：统计每分钟的请求量

```bash
journalctl -u order-service --since "1 hour ago" -o cat --no-pager \
  | jq -r '.time[:16]' \
  | sort | uniq -c | tail -n 20
```

### 场景四：导出慢请求为 CSV

```bash
journalctl -u order-service --since today -o cat --no-pager \
  | jq -r 'select(.latency_ms > 500) | [.time, .uid, .path, .latency_ms, .err // ""] | @csv' \
  > /tmp/slow-requests.csv
```

### 场景五：统计 panic 最多的代码位置

```bash
grep -A 30 "panic" app.log \
  | grep -oE '[a-zA-Z0-9_/]+\.go:[0-9]+' \
  | sort | uniq -c | sort -rn
```

### 场景六：跨日志文件搜索同一个 trace_id

```bash
find /var/log/ -name "*.log" -mtime -1 -print0 \
  | xargs -0 grep "trace_id=abc123def" \
  | sort -t: -k2
```

## 6.8 管道调试技巧

**技巧一：逐步截断**

```bash
grep "ERROR" app.log | head
grep "ERROR" app.log | awk '{print $6}' | head
grep "ERROR" app.log | awk '{print $6}' | sort | uniq -c | head
```

**技巧二：用 tee 保存中间结果**

```bash
grep "ERROR" app.log | tee /tmp/step1.txt | awk '{print $6}' | tee /tmp/step2.txt | sort | uniq -c
```

**技巧三：用 wc -l 确认数据量**

```bash
grep "ERROR" app.log | wc -l
grep "ERROR" app.log | awk '{print $6}' | wc -l
# 两个数应该相同，不同说明 awk 有问题
```

## 6.9 命令速查表

```
语法                           用途
──────────────────────────────────────────────────
cmd1 | cmd2                   管道：cmd1 输出作为 cmd2 输入
cmd | tee file                分流：同时输出到屏幕和文件
cmd | tee -a file             分流追加模式
cmd | xargs CMD               把输入转为 CMD 的参数
cmd | xargs -I {} CMD {}      占位符模式
diff <(cmd1) <(cmd2)          进程替换：对比两个命令输出
cmd > file 2>&1               stdout 和 stderr 都重定向到文件
cmd 2>&1 | grep "err"         把 stderr 也接入管道
```

## 章末思考题

**题目一：`grep "ERROR" app.log | awk '{print $6}' | sort | uniq -c | sort -rn` 这条管道链中，如果日志有 100 万行，ERROR 行有 1 万行。性能瓶颈最可能在哪个环节？有没有优化空间？**

> **参考答案**：瓶颈最可能在第一个 `sort`。grep 和 awk 都是流式处理（读一行处理一行，内存恒定），但 sort 需要把全部 1 万行读入内存才能排序，是整条管道中唯一的"阻塞点"。优化方式：① 如果只看 top N，可以限制数据量；② 超大数据集用 `sort -T 目录 --parallel=N` 多线程排序；③ 用 awk 关联数组一次遍历完成计数：`awk '{count[$1]++} END {for(k in count) print count[k], k}'`，不需要排序。

**题目二：管道中 `grep "ERROR" app.log | jq '.'` 报错 `parse error`，因为部分 ERROR 行不是 JSON（如 panic 堆栈）。怎么让 jq 跳过非 JSON 行？**

> **参考答案**：`grep "ERROR" app.log | jq -R 'fromjson? | select(. != null)'`。`-R` 让 jq 把每行当原始字符串，`fromjson?` 尝试解析，`?` 失败时不报错返回 null，`select(. != null)` 过滤掉失败行。或更简单：`grep "ERROR" app.log | grep "^{" | jq '.'`——只把 `{` 开头的行传给 jq。

**题目三：写一条命令，实时监控日志，ERROR 出现时自动提取错误信息并追加写入独立文件，同时终端显示。**

> **参考答案**：
> ```bash
> tail -F app.log | grep --line-buffered "ERROR" | tee -a /tmp/error-summary.log
> ```
> JSON 日志版本：
> ```bash
> journalctl -u order-service -f -o cat \
>   | jq --unbuffered -r 'select(.level=="ERROR") | [.time, .path, .err] | @tsv' \
>   | tee -a /tmp/error-summary.tsv
> ```
> 关键：`tail -F` 应对轮转、`--line-buffered`/`--unbuffered` 保证实时、`tee -a` 追加写入。
