# 第4章：跨平台能力与高级特性

> **开篇回扣**：前三章聚焦 Web 自动化和架构对比。但 Midscene 的野心不止于浏览器——它的纯视觉路线天然适用于**任何有屏幕的平台**。本章探索 Midscene 的扩展版图：移动端自动化、YAML 声明式脚本、Bridge Mode 调试利器、可视化报告解读、以及如何选择最适合你的 VLM 模型。

---

## 4.1 跨平台能力

### 4.1.1 核心原理

Midscene 的纯视觉路线意味着它不依赖特定平台的 DOM/View Tree——只需要两样东西：
1. **截屏**能力（获取当前界面的图像）
2. **点击/输入**能力（在指定坐标执行操作）

这让同一套 AI API 可以适配完全不同的平台：

```
┌──────────────────────────────────────────┐
│        统一的 Midscene AI API             │
│  aiTap / aiInput / aiQuery / aiAssert    │
└─────────┬──────────┬──────────┬──────────┘
          │          │          │
    ┌─────▼─────┐ ┌──▼───┐ ┌───▼────┐
    │   Web     │ │Android│ │  iOS   │ ...
    │ Playwright│ │  adb  │ │  WDA   │
    │ Puppeteer │ │       │ │        │
    └───────────┘ └───────┘ └────────┘
```

### 4.1.2 Android 自动化

基于 `adb`（Android Debug Bridge），Midscene 可以控制真机或模拟器。

**安装**：
```bash
npm install @midscene/android --save-dev
```

**脚本示例**：
```typescript
import { AgentOnAndroid } from '@midscene/android';

const agent = new AgentOnAndroid();
await agent.connect(); // 连接 adb 设备

// 操作与 Web 完全一致
await agent.aiTap('the Settings icon');
await agent.aiInput('WiFi network name', 'the search field');
await agent.aiAssert('WiFi settings page is displayed');

// Android 专有操作
await agent.launch('com.example.myapp'); // 启动应用
await agent.terminate('com.example.myapp'); // 关闭应用

await agent.destroy();
```

**YAML 方式**：
```yaml
android:
  deviceId: "emulator-5554"  # 可选，指定设备

tasks:
  - name: Open Settings
    flow:
      - launch: com.android.settings
      - aiWaitFor: Settings page is loaded
      - aiTap: the WiFi option
      - aiAssert: WiFi settings are displayed
```

### 4.1.3 iOS 自动化

基于 WebDriverAgent (WDA)，支持真机和模拟器。

**安装**：
```bash
npm install @midscene/ios --save-dev
```

**脚本示例**：
```typescript
import { AgentOnIOS } from '@midscene/ios';

const agent = new AgentOnIOS();
await agent.connect();

await agent.aiTap('the Safari icon');
await agent.aiWaitFor('Safari browser is open');
await agent.aiInput('https://example.com', 'the address bar');

await agent.destroy();
```

### 4.1.4 桌面自动化

Midscene 还支持 Mac/Windows/Linux 桌面应用：

```bash
npm install @midscene/computer --save-dev
```

```typescript
import { AgentOnComputer } from '@midscene/computer';

const agent = new AgentOnComputer();
await agent.aiKeyboardPress(null, { keyName: 'Cmd+Space' }); // Spotlight
await agent.aiInput('Terminal', 'the Spotlight search');
await agent.aiTap('Terminal.app in search results');
```

### 4.1.5 平台支持一览

| 平台 | 包名 | 底层技术 | 成熟度 |
|------|------|---------|--------|
| Web (Playwright) | `@midscene/web` | Playwright CDP | 最成熟 |
| Web (Puppeteer) | `@midscene/web` | Puppeteer CDP | 成熟 |
| Web (Bridge) | `@midscene/web` | Chrome Extension WS | 成熟 |
| Android | `@midscene/android` | adb | 可用 |
| iOS | `@midscene/ios` | WebDriverAgent | 可用 |
| Desktop | `@midscene/computer` | 系统级截屏+输入 | 可用 |
| HarmonyOS | 社区支持 | DevEco 工具链 | 早期 |

## 4.2 YAML 声明式脚本

YAML 模式是 Midscene 的一个独特特性——让不会写 TypeScript 的团队成员也能编写自动化脚本。

### 4.2.1 完整语法

一个 YAML 脚本由三部分组成：

```yaml
# 1. 目标平台配置
web:
  url: https://todomvc.com/examples/react/dist/
  viewportWidth: 1280
  viewportHeight: 960
  # cookie: ./cookies.json           # 可选：注入 Cookie
  # bridgeMode: false                # 可选：Bridge Mode
  # acceptInsecureCerts: false       # 可选：接受不安全证书

# 2. Agent 配置（可选）
agent:
  testId: "todomvc-yaml-test"
  generateReport: true
  aiActContext: "This is a TodoMVC app. The input field is at the top."
  cache:
    id: "todomvc-yaml"
    strategy: "read-write"

# 3. 任务列表
tasks:
  - name: Add todos
    flow:
      - aiInput:
          locate: the new todo input field
          value: "Buy groceries"
      - aiKeyboardPress:
          locate: the new todo input field
          keyName: Enter
      - aiInput:
          locate: the new todo input field
          value: "Cook dinner"
      - aiKeyboardPress:
          locate: the new todo input field
          keyName: Enter

  - name: Verify todos added
    flow:
      - aiAssert: there are exactly 2 todo items on the page
      - aiQuery:
          name: todoList
          dataDemand: "string[], all todo item texts"

  - name: Complete first todo
    flow:
      - aiTap: the checkbox next to "Buy groceries"
      - aiAssert: '"Buy groceries" appears completed'
      - sleep: 500

  - name: Filter and verify
    flow:
      - aiTap: the "Active" filter
      - aiAssert: only 1 todo item is visible
      - aiTap: the "All" filter
```

### 4.2.2 YAML 动作参考

| 动作 | 类型 | 语法示例 |
|------|------|---------|
| `ai` / `aiAct` | 自动规划 | `- ai: Add a todo and mark it done` |
| `aiTap` | 即时操作 | `- aiTap: the login button` |
| `aiHover` | 即时操作 | `- aiHover: the menu item` |
| `aiInput` | 即时操作 | 需要 `locate` + `value` |
| `aiKeyboardPress` | 即时操作 | 需要 `locate` + `keyName` |
| `aiScroll` | 即时操作 | 需要 `locate` + `direction` 等 |
| `aiQuery` | 数据提取 | 需要 `name` + `dataDemand` |
| `aiAssert` | 断言 | `- aiAssert: page title is correct` |
| `aiWaitFor` | 等待 | `- aiWaitFor: results loaded` |
| `sleep` | 暂停 | `- sleep: 2000` |
| `javascript` | JS 执行 | `- javascript: return document.title` |
| `recordToReport` | 调试记录 | `- recordToReport: Before checkout` |

### 4.2.3 高级用法：文件上传与图片提示

```yaml
tasks:
  - name: Upload file
    flow:
      - aiTap: Choose file button
        fileChooserAccept: ./fixtures/document.pdf

  - name: Tap icon using image hint
    flow:
      - aiTap:
          locate:
            prompt: Tap the GitHub logo
            images:
              - name: GitHub logo
                url: https://github.githubassets.com/favicons/favicon.svg
            convertHttpImage2Base64: true
```

### 4.2.4 运行 YAML 脚本

**命令行运行**：
```bash
npx midscene run ./scripts/todomvc-test.yaml
```

**在 JavaScript 中运行**：
```typescript
import { PlaywrightAgent } from '@midscene/web/playwright';
import fs from 'fs';

const yamlContent = fs.readFileSync('./scripts/todomvc-test.yaml', 'utf-8');

// 注意：通过 runYaml 执行时，只解析 tasks 字段
// web 和 agent 配置需要在创建 Agent 时指定
const result = await agent.runYaml(yamlContent);

// result 包含所有 aiQuery 的结果（以 name 为 key）
console.log(result.todoList); // ["Buy groceries", "Cook dinner"]
```

### 4.2.5 YAML vs TypeScript：何时用哪个？

| 场景 | YAML | TypeScript |
|------|------|-----------|
| 快速原型验证 | ✅ 更快 | ❌ 需要项目搭建 |
| QA 团队维护 | ✅ 门槛低 | ❌ 需要编程能力 |
| 复杂条件逻辑 | ❌ 不支持 if/else | ✅ 完全灵活 |
| 数据驱动测试 | ❌ 有限 | ✅ 循环+参数化 |
| CI/CD 集成 | ✅ | ✅ |
| 与 Playwright Test 集成 | ❌ | ✅ 深度集成 |

## 4.3 Bridge Mode — 接管已有浏览器

### 4.3.1 什么是 Bridge Mode

Bridge Mode 让你的 Node.js 脚本通过 WebSocket **控制你桌面上正在运行的 Chrome 浏览器**。

核心价值：**复用已有的登录状态、Cookie、插件、页面上下文**。

适用场景：
- 测试需要登录（SSO / 2FA）的内部系统
- 调试已经在浏览器中打开的页面
- 需要浏览器插件配合的自动化

### 4.3.2 架构

```
┌──────────────────┐     WebSocket      ┌──────────────────┐
│  Node.js 脚本     │ ←──────────────→  │  Chrome 扩展      │
│  AgentOverChrome  │   ws://localhost   │  Midscene 扩展    │
│  Bridge           │      :3766         │  (注入到页面中)    │
└──────────────────┘                     └──────────────────┘
                                                  │
                                         ┌────────▼─────────┐
                                         │  桌面 Chrome      │
                                         │  已登录的页面      │
                                         │  已有的 Cookie     │
                                         │  安装的插件        │
                                         └──────────────────┘
```

### 4.3.3 使用方法

**前置条件**：
1. 安装 Midscene Chrome 扩展
2. 扩展图标显示**黄色圆点**（表示正在监听 WebSocket 连接）

**TypeScript 脚本**：

```typescript
import { AgentOverChromeBridge } from '@midscene/web/bridge-mode';

async function main() {
  const agent = new AgentOverChromeBridge();

  // 方式1：在新标签页中打开 URL
  await agent.connectNewTabWithUrl('https://your-internal-app.com');

  // 方式2：接管当前标签页
  // await agent.connectCurrentTab();

  // 正常使用 Midscene API
  await agent.aiTap('the user profile dropdown');
  await agent.aiAssert('user is logged in as admin');

  const userData = await agent.aiQuery<{ name: string; role: string }>(
    '{name: string, role: string}, the logged-in user info from the header'
  );
  console.log(userData);

  await agent.destroy();
}

main();
```

**YAML 方式**：
```yaml
web:
  url: https://your-internal-app.com
  bridgeMode: newTabWithUrl              # 或 'currentTab'
  closeNewTabsAfterDisconnect: true      # 断开时关闭新开的标签页

tasks:
  - name: Check login status
    flow:
      - aiAssert: user is logged in
      - aiQuery:
          name: userInfo
          dataDemand: "{name: string, role: string}, the user info"
```

### 4.3.4 远程访问

Bridge Mode 默认只监听 `localhost`，但可以配置远程访问：

```typescript
// 允许远程连接（绑定 0.0.0.0）
const agent = new AgentOverChromeBridge({ allowRemoteAccess: true });

// 或指定具体地址
const agent = new AgentOverChromeBridge({
  host: '192.168.1.100',
  port: 3766,
});
```

### 4.3.5 Bridge Mode 的限制

以下配置在 Bridge Mode 中**不生效**（由桌面浏览器设置控制）：
- `userAgent`
- `viewportWidth` / `viewportHeight`
- `deviceScaleFactor`
- `waitForNetworkIdle`
- `cookie`

## 4.4 可视化调试报告

### 4.4.1 报告内容

Midscene 自动生成的 HTML 报告是调试的核心工具。每份报告包含：

| 信息 | 说明 |
|------|------|
| 执行时间线 | 每个 AI 操作的开始/结束时间 |
| 截图序列 | 每步操作前后的页面截图（JPEG） |
| VLM 定位结果 | 元素在截图中的坐标标注 |
| 操作规划 | `aiAct` 的分步规划详情 |
| 缓存命中 | 标记哪些操作命中了缓存 |
| 断言结果 | 每个 `aiAssert` 的通过/失败 |
| 错误详情 | 失败时的完整错误信息和上下文 |

### 4.4.2 报告格式选择

```typescript
// 单 HTML 文件（默认，截图 base64 内嵌，便于分享）
const agent = new PlaywrightAgent(page, {
  outputFormat: 'single-html',
});

// HTML + 外部资源（截图独立存放，适合大量测试）
const agent = new PlaywrightAgent(page, {
  outputFormat: 'html-and-external-assets',
});
// 注意：此格式需要 HTTP 服务器才能查看，不能直接打开
```

### 4.4.3 报告工具

Midscene v1.7.0+ 提供命令行工具处理报告：

```bash
# 提取报告为 JSON + 独立截图
npx @midscene/web report-tool --action split \
  --htmlPath ./midscene_run/report/test-report.html \
  --outputDir ./report-output

# 转换为 Markdown（适合归档或文档化）
npx @midscene/web report-tool --action to-markdown \
  --htmlPath ./midscene_run/report/test-report.html \
  --outputDir ./report-output
```

### 4.4.4 合并多个报告

在 Playwright Test 集成中，可以用 reporter 自动合并：

```typescript
// playwright.config.ts
reporter: [
  ['@midscene/web/playwright-reporter', {
    type: 'merged',         // 所有测试合并为一个报告
    // type: 'separate',    // 每个测试独立报告
  }],
],
```

手动合并：
```typescript
import { ReportMergingTool } from '@midscene/core/report';

const tool = new ReportMergingTool();
tool.append({
  reportFilePath: agent1.reportFile,
  reportAttributes: { testId: 'test-1', testTitle: 'Login Test', testStatus: 'passed' },
});
tool.append({
  reportFilePath: agent2.reportFile,
  reportAttributes: { testId: 'test-2', testTitle: 'Checkout Test', testStatus: 'failed' },
});
tool.mergeReports('combined-report', { rmOriginalReports: true });
```

## 4.5 模型选择策略

### 4.5.1 基础配置

```bash
# 必填四项
export MIDSCENE_MODEL_API_KEY="sk-..."
export MIDSCENE_MODEL_BASE_URL="https://xxx/v1"    # 不要追加 /chat/completion
export MIDSCENE_MODEL_NAME="qwen3-vl-plus"
export MIDSCENE_MODEL_FAMILY="qwen3-vl"             # 决定坐标处理方式
```

`MIDSCENE_MODEL_FAMILY` 非常重要——不同模型族的坐标输出格式不同，设错会导致定位偏移。

### 4.5.2 模型推荐

| 模型 | 评级 | 特点 | 适合场景 |
|------|------|------|---------|
| **Doubao Seed 1.6/2.0** | ⭐⭐⭐⭐ | UI 规划能力强，速度稍慢 | 复杂操作流程 |
| **Qwen3-VL** | ⭐⭐⭐⭐ | 性价比高，综合能力强 | 通用场景首选 |
| **Zhipu GLM-4.6V** | 新模型 | 开源权重可用 | 私有化部署 |
| **Gemini 3 Flash** | ⭐⭐⭐ | 速度快但价格较高 | 对延迟敏感的场景 |
| **GPT-5** | 不推荐作默认 | 元素定位能力弱 | 可作为规划模型 |
| **UI-TARS** | 专用模型 | 专为 UI 任务训练 | 高精度需求 |

### 4.5.3 多模型策略

Midscene 支持为不同用途配置不同模型：

| 角色 | 用途 | 环境变量前缀 |
|------|------|------------|
| Default（默认） | 元素定位 | `MIDSCENE_MODEL_*` |
| Planning（规划） | `aiAct` 的步骤规划 | `MIDSCENE_PLANNING_MODEL_*` |
| Insight（洞察） | `aiQuery`/`aiAssert` 等数据提取和判断 | `MIDSCENE_INSIGHT_MODEL_*` |

```bash
# 默认模型：Qwen3-VL（擅长定位）
export MIDSCENE_MODEL_NAME="qwen3-vl-plus"
export MIDSCENE_MODEL_FAMILY="qwen3-vl"

# 规划模型：Doubao（擅长复杂规划）
export MIDSCENE_PLANNING_MODEL_NAME="doubao-seed-1.6-vision"
export MIDSCENE_PLANNING_MODEL_FAMILY="doubao-vision"

# 洞察模型：可以用较便宜的模型
export MIDSCENE_INSIGHT_MODEL_NAME="qwen3-vl-lite"
export MIDSCENE_INSIGHT_MODEL_FAMILY="qwen3-vl"
```

也可以在代码中为单个 Agent 配置：

```typescript
const agent = new PlaywrightAgent(page, {
  modelConfig: {
    MIDSCENE_MODEL_NAME: 'qwen3-vl-plus',
    MIDSCENE_MODEL_FAMILY: 'qwen3-vl',
    MIDSCENE_PLANNING_MODEL_NAME: 'doubao-seed-1.6-vision',
    MIDSCENE_PLANNING_MODEL_FAMILY: 'doubao-vision',
    MIDSCENE_PLANNING_MODEL_BASE_URL: 'https://ark.cn-beijing.volces.com/api/v3',
    MIDSCENE_PLANNING_MODEL_API_KEY: 'sk-doubao-key',
  },
});
```

### 4.5.4 其他模型配置项

| 环境变量 | 默认值 | 说明 |
|---------|--------|------|
| `MIDSCENE_MODEL_TIMEOUT` | 180000ms | API 超时 |
| `MIDSCENE_MODEL_MAX_TOKENS` | 2048 | 最大响应 token |
| `MIDSCENE_MODEL_RETRY_COUNT` | 1 | 失败重试次数 |
| `MIDSCENE_MODEL_REASONING_ENABLED` | 未设置 | 开启思维链模式 |
| `MIDSCENE_MODEL_HTTP_PROXY` | 未设置 | HTTP 代理 |
| `MIDSCENE_PREFERRED_LANGUAGE` | 自动（GMT+8 时中文） | 响应语言 |
| `MIDSCENE_RUN_DIR` | `midscene_run` | 报告/日志目录 |

### 4.5.5 性能调优

```typescript
const agent = new PlaywrightAgent(page, {
  // 减少截图尺寸（2 = 宽高各缩小一半），降低 Token 消耗
  screenshotShrinkFactor: 2,

  // 缩短操作间等待时间（默认 300ms）
  waitAfterAction: 100,

  // 缓存加速
  cache: { id: 'perf-test', strategy: 'read-write' },
});
```

## 4.6 实战产出：YAML 声明式脚本

以下是本章的实战产出——一个完整的 TodoMVC YAML 自动化脚本：

```yaml
# todomvc-full-test.yaml
# TodoMVC 完整功能测试（YAML 声明式）

web:
  url: https://todomvc.com/examples/react/dist/
  viewportWidth: 1280
  viewportHeight: 960

agent:
  testId: "todomvc-yaml-full"
  generateReport: true
  aiActContext: >
    This is a TodoMVC application built with React.
    The input field for new todos is at the top of the page.
    Each todo item has a checkbox on the left and a delete button (×) that appears on hover.
  cache:
    id: "todomvc-yaml-cache"
    strategy: "read-write"

tasks:
  # ===== 创建待办 =====
  - name: Add first todo
    flow:
      - aiInput:
          locate: the new todo input field
          value: "Buy groceries"
      - aiKeyboardPress:
          locate: the new todo input field
          keyName: Enter
      - recordToReport: Added first todo

  - name: Add second todo
    flow:
      - aiInput:
          locate: the new todo input field
          value: "Read a book"
      - aiKeyboardPress:
          locate: the new todo input field
          keyName: Enter

  - name: Add third todo
    flow:
      - aiInput:
          locate: the new todo input field
          value: "Go for a walk"
      - aiKeyboardPress:
          locate: the new todo input field
          keyName: Enter

  # ===== 验证创建 =====
  - name: Verify all todos created
    flow:
      - aiAssert: there are exactly 3 todo items on the page
      - aiQuery:
          name: allTodos
          dataDemand: "string[], all todo item texts"
      - recordToReport: Verified 3 todos

  # ===== 完成操作 =====
  - name: Complete first todo
    flow:
      - aiTap: the checkbox next to "Buy groceries"
      - aiAssert: '"Buy groceries" appears completed with strikethrough'
      - sleep: 300

  # ===== 过滤验证 =====
  - name: Filter active items
    flow:
      - aiTap: the "Active" filter link
      - aiAssert: only 2 todo items are visible
      - aiQuery:
          name: activeTodos
          dataDemand: "string[], all visible todo item texts"
      - recordToReport: Active filter applied

  - name: Filter completed items
    flow:
      - aiTap: the "Completed" filter link
      - aiAssert: only 1 todo item is visible showing "Buy groceries"
      - recordToReport: Completed filter applied

  - name: Show all items
    flow:
      - aiTap: the "All" filter link
      - aiAssert: all 3 todo items are visible

  # ===== 清除已完成 =====
  - name: Clear completed
    flow:
      - aiTap: the "Clear completed" button
      - aiAssert: there are exactly 2 todo items remaining
      - aiAssert: '"Buy groceries" is no longer in the list'
      - aiQuery:
          name: finalTodos
          dataDemand: "string[], all remaining todo item texts"
      - recordToReport: Test complete - final state
```

**运行**：
```bash
npx midscene run ./todomvc-full-test.yaml
```

运行后查看 `midscene_run/report/` 中的 HTML 报告，你可以看到每一步的截图、VLM 响应和缓存命中情况。

## 思考题

### Q1：YAML 脚本不支持条件判断（if/else）和循环（for），这限制了哪些场景？有没有变通方案？

**答案**：

**受限场景**：
- 数据驱动测试（遍历一组测试数据）
- 条件分支（"如果弹窗出现就关闭，否则继续"）
- 错误恢复（"如果步骤失败就执行备用路径"）

**变通方案**：

1. **用 `aiAct` 自动规划处理条件逻辑**：
```yaml
- ai: "If there is a cookie consent popup, click 'Accept'. Otherwise, do nothing."
```

### Q2：Bridge Mode 的最大价值是"复用登录状态"。如果你的被测系统需要 SSO + 2FA，如何设计自动化方案？

**答案**：

SSO + 2FA 是自动化测试的经典难题。Bridge Mode 的解决方案：

**手动登录 + 自动化测试分离**：
1. 人工在桌面 Chrome 中完成 SSO + 2FA 登录
2. Bridge Mode 接管已登录的浏览器会话
3. 自动化脚本在已认证的会话中执行测试
4. Cookie/Token 有效期内无需重新登录

```typescript
const agent = new AgentOverChromeBridge();
// 接管当前已登录的标签页
await agent.connectCurrentTab();
// 此时 Cookie、Session 都在，无需处理登录
await agent.aiTap('the Admin Dashboard link');
await agent.aiAssert('admin dashboard is loaded');
```

**自动化 + 手动混合流程**：
1. 脚本导航到登录页
2. 暂停执行，等待人工完成 2FA
3. 人工完成后，脚本继续执行

这种"人机协作"模式是 Bridge Mode 的独特优势，纯 headless 方案无法实现。

### Q3：多模型策略中，为什么定位和规划要用不同的模型？这背后的 AI 原理是什么？

**答案**：

不同的 VLM 在不同子任务上的能力差异显著：

**元素定位**（Default 模型）需要：
- 精确的视觉感知（"这个按钮在哪个像素位置"）
- 快速响应（每步操作都要调用）
- 较低的 Token 消耗

**操作规划**（Planning 模型）需要：
- 强大的推理能力（"要完成这个任务，应该先做什么再做什么"）
- 理解复杂指令的能力
- 不需要特别快（只在 `aiAct` 时调用一次）

**数据提取/断言**（Insight 模型）需要：
- 准确的文本识别和理解
- 结构化输出能力
- 对于简单断言，可以用较便宜的模型

这就像一个团队：定位模型是"眼睛"（看得准），规划模型是"大脑"（想得清），洞察模型是"分析师"（提得对）。不同岗位用不同人才，整体效率最高。
