# 第1章：Midscene 核心理念与快速上手

> **开篇回扣**：作为有 Playwright 实战经验的开发者，你一定经历过这样的痛苦——精心编写的 `page.locator('.btn-primary')` 在某次 UI 重构后全军覆没。CSS 类名改了、DOM 层级变了、甚至按钮文案都换了。你不得不逐个修复选择器，而这些修复和业务逻辑毫无关系。Midscene 要解决的，正是这个根本性问题。

---

## 1.1 两种范式：选择器驱动 vs 视觉驱动

### 选择器驱动（Playwright / Selenium）

你已经非常熟悉这种模式：

```typescript
// Playwright 典型写法
await page.locator('#todo-input').fill('Buy groceries');
await page.locator('.toggle').first().click();
await expect(page.locator('.todo-count')).toHaveText('0 items left');
```

这段代码的**隐含假设**是：
1. 输入框的 `id` 是 `todo-input`
2. 勾选按钮的 class 是 `toggle`
3. 计数器的 class 是 `todo-count`

任何一个假设失效，测试就断了。本质上，你在用**程序语言**（CSS 选择器、XPath）去描述一个**视觉概念**（"那个输入框"、"那个勾选按钮"）。

### 视觉驱动（Midscene）

```typescript
// Midscene 写法
await agent.aiInput('Buy groceries', 'the todo input box');
await agent.aiTap('the checkbox of "Buy groceries"');
await agent.aiAssert('the remaining count shows 0 items');
```

没有选择器。Midscene 在每一步操作时：
1. 对当前页面截图
2. 将截图发送给 VLM（视觉语言模型）
3. VLM 根据你的自然语言描述，在截图中定位目标元素
4. 返回元素坐标，Midscene 执行操作

这不是"换了一种定位方式"。这是**把自动化的抽象层级从 DOM 结构提升到了视觉语义**。

### 关键区别一览

| 维度 | 选择器驱动 | 视觉驱动（Midscene） |
|------|-----------|---------------------|
| 定位依据 | DOM 结构（id/class/xpath） | 页面截图 + 自然语言 |
| 抗 UI 变更能力 | 弱——选择器与 DOM 强耦合 | 强——只要视觉语义不变 |
| 执行速度 | 快（直接 DOM 操作） | 较慢（每步需 VLM 推理） |
| 确定性 | 100% 确定 | 概率性（依赖模型能力） |
| 成本 | 无额外成本 | VLM API 调用产生 Token 费用 |
| 调试体验 | 浏览器 DevTools | 可视化回放报告 |

> **核心洞察**：Midscene 不是 Playwright 的替代品，而是在它之上叠加了一层视觉理解能力。实际上，Midscene 的 Web 自动化底层就是 Playwright 或 Puppeteer。

## 1.2 Midscene 技术架构速览

```
┌─────────────────────────────────────────┐
│              你的测试代码                  │
│   agent.aiTap('login button')           │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│          Midscene SDK                    │
│  @midscene/web (Playwright/Puppeteer)   │
│  - 截图采集                               │
│  - 坐标映射                               │
│  - 缓存管理                               │
│  - 可视化报告生成                          │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│           VLM 视觉语言模型                │
│  Qwen3-VL / Doubao / Gemini / UI-TARS  │
│  - 截图理解                               │
│  - 元素定位                               │
│  - 操作规划（Auto Planning）              │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│        浏览器引擎                         │
│  Playwright / Puppeteer / Chrome CDP    │
│  - 实际页面操作执行                       │
└─────────────────────────────────────────┘
```

关键点：
- **纯视觉路线（v1.0+）**：所有元素定位和交互基于截图，不依赖 DOM 提取（数据提取场景可选启用 DOM 辅助）
- **VLM 是可替换的**：支持多种模型，通过环境变量配置
- **底层复用成熟引擎**：不重复造轮子，专注于视觉理解层

## 1.3 环境搭建

### 方式一：Chrome 扩展（零代码体验）

这是最快的上手方式，让你在 5 分钟内感受 Midscene 的能力。

**步骤 1：安装扩展**

Chrome Web Store 搜索 "Midscene" 或直接访问：
```
https://chromewebstore.google.com/detail/midscene/gbldofcpkknbggpkmbdaefngejllnief
```

**步骤 2：配置模型**

打开扩展设置，填入 VLM 配置（以 Qwen3-VL 为例）：

| 配置项 | 值 |
|--------|-----|
| `MIDSCENE_MODEL_BASE_URL` | `https://dashscope.aliyuncs.com/compatible-mode/v1` |
| `MIDSCENE_MODEL_API_KEY` | 你的阿里云 API Key |
| `MIDSCENE_MODEL_NAME` | `qwen3-vl-plus` |
| `MIDSCENE_MODEL_FAMILY` | `qwen3-vl` |

> **注意**：`BASE_URL` 不要追加 `/chat/completion`，Midscene 会自动拼接。

**步骤 3：开始体验**

打开任意网页，点击 Midscene 扩展图标，侧边栏提供四个标签页：

| 标签页 | 对应 API | 用途 |
|--------|----------|------|
| **Act** | `aiAct` | 自然语言描述多步操作 |
| **Query** | `aiQuery` | 从页面提取结构化数据 |
| **Assert** | `aiAssert` | 断言页面状态 |
| **Tap** | `aiTap` | 点击指定元素 |

## 1.4 第一个 Midscene 脚本

### Chrome 扩展体验

1. 打开 https://todomvc.com/examples/react/dist/
2. 点击 Midscene 扩展，在 **Act** 标签页输入：
   ```
   在输入框中输入 "Learn Midscene"，然后按回车
   ```
3. 点击执行，观察 Midscene 如何：
   - 截取当前页面
   - 发送给 VLM 分析
   - 定位输入框并执行操作

4. 切换到 **Query** 标签页，输入：
   ```json
   { "todos": "string[], 所有待办事项的文本" }
   ```
5. 观察返回的结构化数据

6. 切换到 **Assert** 标签页，输入：
   ```
   页面上显示有一个待办事项 "Learn Midscene"
   ```

### SDK 脚本

创建 `demo.ts`：

```typescript
import { chromium } from 'playwright';
import { PlaywrightAgent } from '@midscene/web/playwright';

async function main() {
  // 启动浏览器
  const browser = await chromium.launch({ headless: false }); // 先用有头模式，方便观察
  const page = await browser.newPage();

  // 访问 TodoMVC
  await page.goto('https://todomvc.com/examples/react/dist/');
  await page.waitForLoadState('networkidle');

  // 创建 Midscene Agent
  const agent = new PlaywrightAgent(page, {
    generateReport: true, // 生成可视化报告
  });

  // --- 交互：添加待办事项 ---
  await agent.aiInput('Learn Midscene', 'the todo input box');
  await agent.aiKeyboardPress('the todo input box', { keyName: 'Enter' });

  await agent.aiInput('Write E2E tests', 'the todo input box');
  await agent.aiKeyboardPress('the todo input box', { keyName: 'Enter' });

  // --- 数据提取：获取所有待办事项 ---
  const todos = await agent.aiQuery<string[]>(
    'string[], all todo item texts on the page'
  );
  console.log('Current todos:', todos);

  // --- 断言：验证页面状态 ---
  await agent.aiAssert('there are exactly 2 todo items on the page');
  await agent.aiAssert('the item counter shows "2 items left"');

  // --- 交互：完成一个事项 ---
  await agent.aiTap('the checkbox next to "Learn Midscene"');

  // --- 再次断言 ---
  await agent.aiAssert('the item counter shows "1 item left"');
  await agent.aiAssert('"Learn Midscene" appears to be completed (has strikethrough)');

  console.log('All assertions passed!');
  console.log('Report generated at:', agent.reportFile);

  await browser.close();
}

main().catch(console.error);
```

运行：

```bash
npx tsx demo.ts
```

执行完成后，打开 `midscene_run/report/` 目录下生成的 HTML 报告，你会看到每一步的截图、VLM 的定位结果、执行耗时等详细信息。

## 1.5 与 Playwright 原生方案的初步对比

同样的操作，Playwright 原生写法：

```typescript
import { chromium } from 'playwright';

async function main() {
  const browser = await chromium.launch({ headless: false });
  const page = await browser.newPage();
  await page.goto('https://todomvc.com/examples/react/dist/');

  // 添加待办
  const input = page.locator('.new-todo');
  await input.fill('Learn Midscene');
  await input.press('Enter');
  await input.fill('Write E2E tests');
  await input.press('Enter');

  // 验证
  await expect(page.locator('.todo-list li')).toHaveCount(2);
  await expect(page.locator('.todo-count')).toContainText('2 items left');

  // 完成一个
  await page.locator('.todo-list li').first().locator('.toggle').click();
  await expect(page.locator('.todo-count')).toContainText('1 item left');

  await browser.close();
}
```

**直观感受**：
- Playwright 版本更快、更确定，但依赖 `.new-todo`、`.todo-list`、`.toggle` 等选择器
- Midscene 版本几乎不需要了解 DOM 结构，但每步操作都需要 VLM 推理，速度更慢
- 如果 TodoMVC 的实现从 React 换成 Vue（DOM 结构变化），Playwright 版本大概率需要修改选择器，而 Midscene 版本很可能无需改动

## 1.6 两种自动化范式

Midscene 提供两种风格的自动化，理解它们的区别至关重要：

### Auto Planning（自动规划）

用 `aiAct` / `ai` 方法，一句自然语言描述整个操作流程，由 VLM 自动拆解步骤：

```typescript
await agent.aiAct('添加三个待办事项：买菜、做饭、洗碗，然后把"买菜"标记为完成');
```

VLM 会自主规划：输入 → 回车 → 输入 → 回车 → 输入 → 回车 → 点击勾选框。

**优点**：代码极简，接近人类描述
**缺点**：VLM 规划可能不稳定，复杂流程容易出错，默认最多 20 次规划循环

### Workflow Style（工作流风格）

开发者控制每一步，AI 只负责定位和理解：

```typescript
const items = ['买菜', '做饭', '洗碗'];
for (const item of items) {
  await agent.aiInput(item, 'the todo input box');
  await agent.aiKeyboardPress('the todo input box', { keyName: 'Enter' });
}
await agent.aiTap('the checkbox next to "买菜"');
```

**优点**：流程可控、可预测、更稳定
**缺点**：代码量更多，不如 Auto Planning 简洁

> **实践建议**：正式测试中优先使用 Workflow Style，Auto Planning 适合原型验证和简单探索场景。

## 思考题

### Q1：Midscene 的纯视觉路线意味着什么？如果页面上有两个外观完全一样的按钮，Midscene 能区分吗？

**答案**：

纯视觉路线意味着 Midscene（v1.0+）的所有元素定位和交互操作**完全基于页面截图**，不依赖 DOM 结构。这带来的直接影响：

- **优势**：不受 DOM 变更影响，同样的视觉布局 = 同样的测试行为
- **挑战**：如果两个按钮在视觉上完全一样（相同文案、相同样式、相同大小），VLM 确实可能无法区分

应对策略：
1. **提供更精确的上下文描述**：不说"点击提交按钮"，而说"点击表单底部的提交按钮"或"点击右侧面板中的提交按钮"
2. **使用 `deepLocate` 选项**：开启后 Midscene 会调用两次 VLM，第二次专注于精确定位，对小元素或相似元素效果更好
3. **结合位置信息**：描述元素的相对位置，利用 VLM 的空间理解能力

```typescript
// 不够精确
await agent.aiTap('the submit button');

// 更精确
await agent.aiTap('the submit button at the bottom of the form');

// 开启 deepLocate
await agent.aiTap('the submit button', { deepLocate: true });
```

### Q2：Midscene 的 VLM 调用会产生 Token 费用。一个包含 50 个测试用例的套件，粗略估算成本量级是多少？

**答案**：

粗略估算框架：

- 每次 VLM 调用 ≈ 一张截图（约 1000-2000 token 输入）+ 响应（约 200-500 token）
- 每个测试用例平均 5-10 次 AI 调用（交互 + 断言 + 数据提取）
- 50 个测试用例 ≈ 250-500 次 VLM 调用

以 Qwen3-VL-Plus 为例（阿里云定价）：
- 输入约 ¥0.003/千 token，输出约 ¥0.006/千 token
- 每次调用成本约 ¥0.003-0.01
- 50 个用例单次运行 ≈ ¥1-5

**成本控制手段**：
1. **缓存机制**：开启 `cache: { id: "test-suite", strategy: "read-write" }` 后，AI 的操作规划和元素定位会被缓存，重复运行时直接复用，可减少 ~50% 的 VLM 调用
2. **`screenshotShrinkFactor`**：设为 2 可将截图尺寸减半，减少输入 token
3. **Workflow Style 优于 Auto Planning**：Auto Planning 可能触发多轮规划，Token 消耗更高
4. **分层策略**：核心流程用 Midscene（视觉稳定性），简单验证仍用 Playwright 原生断言

### Q3：为什么 Midscene 选择"不缓存"aiAssert 和 aiQuery 的结果？这个设计决策背后的原因是什么？

**答案**：

这是一个深思熟虑的设计决策：

- **`aiAct` 的规划可以缓存**：因为"在输入框输入文字并回车"这个操作步骤序列，在 UI 没有大变的情况下是稳定的
- **`aiTap` 的定位可以缓存**：XPath 路径在页面结构稳定时可复用
- **`aiAssert` 和 `aiQuery` 不能缓存**：因为它们的结果取决于**当前页面的实时状态**

举例：如果你缓存了 `aiAssert('购物车中有 3 件商品')` 的结果（true），那下次运行时即使购物车是空的，也会返回 true——这就完全失去了断言的意义。

同理，`aiQuery` 提取的数据（如商品列表、价格信息）每次运行都可能不同，缓存毫无意义。

**核心原则**：缓存"怎么做"（操作计划），不缓存"看到了什么"（状态判断）。
