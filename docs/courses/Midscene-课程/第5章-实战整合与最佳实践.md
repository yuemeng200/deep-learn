## 5.1 完整项目结构回顾

一个集成了 Midscene 的 Playwright 测试项目的推荐结构：

```
my-project/
├── src/                          # 应用源码
├── e2e/                          # E2E 测试
│   ├── fixture.ts                # Midscene + Playwright 联合 fixture
│   ├── todomvc.spec.ts           # Midscene 驱动的测试用例
│   ├── checkout.spec.ts          # 混合 Midscene + 原生 Playwright
│   └── legacy/                   # 原有的纯 Playwright 用例
│       └── auth.spec.ts
├── scripts/                      # YAML 自动化脚本
│   ├── smoke-test.yaml           # 冒烟测试
│   └── data-extraction.yaml      # 数据采集
├── playwright.config.ts          # Playwright + Midscene reporter 配置
├── .env                          # VLM 配置（不提交到 Git）
├── .env.example                  # VLM 配置模板
├── midscene_run/                 # Midscene 输出（gitignore）
│   ├── report/                   # HTML 可视化报告
│   └── cache/                    # 缓存文件
└── package.json
```

### 关键配置文件

**`playwright.config.ts`**：

```typescript
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './e2e',

  // Midscene 测试需要更长的超时
  timeout: 120 * 1000,
  expect: { timeout: 30 * 1000 },

  // 重试策略：AI 驱动的测试偶尔会失败，合理的重试是必要的
  retries: process.env.CI ? 2 : 0,

  // 并行度：受 VLM API 速率限制，不宜过高
  workers: process.env.CI ? 2 : 1,

  use: {
    baseURL: process.env.BASE_URL || 'http://localhost:3000',
    viewport: { width: 1280, height: 720 },
    screenshot: 'only-on-failure',
    trace: 'retain-on-failure',
  },

  reporter: [
    ['list'],
    ['html', { open: 'never' }], // Playwright 原生报告
    ['@midscene/web/playwright-reporter', {
      type: 'merged',
      outputFormat: 'single-html',
    }], // Midscene 报告
  ],

  projects: [
    {
      name: 'chromium',
      use: {
        browserName: 'chromium',
        // 如果 CI 中截图 DPR 不一致，强制设置
        deviceScaleFactor: 1,
      },
    },
  ],
});
```

**`.env.example`**：


## 5.2 CI/CD 集成

### 5.2.1 GitHub Actions 示例

```yaml
# .github/workflows/e2e.yml
name: E2E Tests

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  e2e-test:
    runs-on: ubuntu-latest
    timeout-minutes: 30

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Install Playwright browsers
        run: npx playwright install chromium --with-deps

      - name: Run E2E tests
        env:
          MIDSCENE_MODEL_BASE_URL: ${{ secrets.MIDSCENE_MODEL_BASE_URL }}
          MIDSCENE_MODEL_API_KEY: ${{ secrets.MIDSCENE_MODEL_API_KEY }}
          MIDSCENE_MODEL_NAME: ${{ secrets.MIDSCENE_MODEL_NAME }}
          MIDSCENE_MODEL_FAMILY: ${{ secrets.MIDSCENE_MODEL_FAMILY }}
          # CI 环境的字体可能导致截图超时
          PW_TEST_SCREENSHOT_NO_FONTS_READY: 1
        run: npx playwright test

      - name: Upload Playwright report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: playwright-report
          path: playwright-report/

      - name: Upload Midscene report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: midscene-report
          path: midscene_run/report/

      - name: Upload Midscene cache
        if: success()
        uses: actions/upload-artifact@v4
        with:
          name: midscene-cache
          path: midscene_run/cache/
```

### 5.2.2 CI 环境注意事项

| 问题 | 解决方案 |
|------|---------|
| **字体渲染差异** | 设置 `PW_TEST_SCREENSHOT_NO_FONTS_READY=1` |
| **DPR 不一致** | 在 Playwright config 中设置 `deviceScaleFactor: 1` |
| **VLM API 超时** | 增大 `MIDSCENE_MODEL_TIMEOUT`（如 300000ms） |
| **网络不稳定** | 设置 `MIDSCENE_MODEL_RETRY_COUNT=2` |
| **并行冲突** | 控制 `workers` 数量，避免 VLM API 速率限制 |
| **缓存失效** | 在 CI 中持久化 `midscene_run/cache/` |

### 5.2.3 缓存持久化策略

CI 中每次运行都是新环境，缓存默认会丢失。两种持久化方案：

**方案1：GitHub Actions Cache**

```yaml
- name: Restore Midscene cache
  uses: actions/cache@v4
  with:
    path: midscene_run/cache
    key: midscene-cache-${{ hashFiles('e2e/**/*.spec.ts') }}
    restore-keys: |
      midscene-cache-

- name: Run tests
  run: npx playwright test

- name: Save Midscene cache
  if: always()
  uses: actions/cache/save@v4
  with:
    path: midscene_run/cache
    key: midscene-cache-${{ hashFiles('e2e/**/*.spec.ts') }}
```

**方案2：将缓存文件提交到仓库**

```bash
# 将缓存文件纳入版本控制
git add midscene_run/cache/*.cache.yaml
git commit -m "chore: update midscene cache"
```

适用场景：UI 变更不频繁，缓存稳定。缺点是仓库体积增长。

### 5.2.4 成本监控

在 CI 中追踪 Token 消耗：

```bash
# 开启性能日志
export DEBUG=midscene:ai:profile:stats

# 运行测试后，日志中会包含每次 VLM 调用的 token 统计
npx playwright test 2>&1 | grep "profile:stats"
```

也可以接入 LangSmith 或 Langfuse 进行长期监控。

## 5.3 Midscene 的能力边界与局限性

这是整个课程最关键的部分——知道一个工具**不能**做什么，比知道它**能**做什么更重要。

### 5.3.1 固有局限

| 局限 | 说明 | 影响程度 |
|------|------|---------|
| **概率性结果** | VLM 的识别/判断不是 100% 确定的 | 高 |
| **速度开销** | 每步操作需要 VLM 推理，比原生操作慢 10-50 倍 | 中 |
| **Token 成本** | 大规模测试套件的持续运行成本 | 中 |
| **不可见元素** | 视觉上不可见的 DOM 元素无法操作 | 中 |
| **动态内容** | 快速变化的内容（动画、实时数据流）难以可靠截取 | 中 |
| **精确像素操作** | VLM 的坐标精度有限，不适合像素级精确操作 | 低 |
| **键盘组合键** | `aiKeyboardPress` 不支持组合键 | 低 |
| **原生对话框** | 右键菜单、文件选择器等原生 UI 需要特殊处理 | 低 |

### 5.3.2 场景适配分析

**Midscene 表现优秀的场景**：
- 表单填写和提交（输入框、下拉框、按钮）
- 导航流程（点击菜单、切换页面）
- 数据验证（从页面提取数据并断言）
- 视觉布局验证（元素是否在正确位置）
- 跨框架/跨版本的通用测试

**Midscene 表现一般的场景**：
- 拖拽操作（`aiPinch` 支持有限）
- Canvas/WebGL 内容（截图可能信息不足）
- 实时流式更新的页面（截图时机不确定）
- 需要精确时序的操作（如动画完成后立即操作）

**Midscene 不适合的场景**：
- 性能测试（VLM 延迟会干扰性能指标）
- 大规模压力测试（Token 成本不可控）
- 需要 100% 确定性的金融/医疗关键路径
- 纯 API 测试（不涉及 UI）

### 5.3.3 常见陷阱与应对

**陷阱 1：过度依赖 Auto Planning**

```typescript
// ❌ 把太多逻辑交给 VLM，容易出错
await agent.aiAct(`
  Login with admin/admin123,
  navigate to user management,
  create a new user with name "Test" and email "test@example.com",
  assign admin role,
  save and verify the user appears in the list
`);

// ✅ 拆分为可控的步骤
await agent.aiInput('admin', 'username field');
await agent.aiInput('admin123', 'password field');
await agent.aiTap('login button');
await agent.aiWaitFor('dashboard is loaded');
await agent.aiTap('User Management menu item');
// ...
```

**陷阱 2：断言描述不够精确**

```typescript
// ❌ 模糊的断言，VLM 可能误判
await agent.aiAssert('the page looks correct');

// ✅ 具体的断言
await agent.aiAssert('the user table shows exactly 5 rows');
await agent.aiAssert('the success toast message contains "User created"');
```

**陷阱 3：忽视缓存管理**

```typescript
// ❌ 不使用缓存，每次都调 VLM
const agent = new PlaywrightAgent(page);

// ✅ 为每个测试套件设置缓存
const agent = new PlaywrightAgent(page, {
  cache: { id: `user-mgmt-${testSuiteName}`, strategy: 'read-write' },
});
```

**陷阱 4：并行过多导致 API 限流**

```typescript
// ❌ 太多 worker 同时调 VLM API
// playwright.config.ts: workers: 10

// ✅ 根据 VLM API 限制调整
// playwright.config.ts: workers: 2
```

## 5.4 决策框架：何时用什么方案

### 5.4.1 三层决策模型

```
                 你的测试需求
                      │
        ┌─────────────┼──────────────┐
        ▼             ▼              ▼
   确定性要求高    抗 UI 变更     探索性/一次性
   速度敏感       需要调试报告    跨系统协调
        │             │              │
        ▼             ▼              ▼
  Playwright 原生   Midscene     Playwright+MCP
```

### 5.4.2 详细决策矩阵

| 因素 | 选 Playwright 原生 | 选 Midscene | 选 Playwright+MCP |
|------|-------------------|-------------|-------------------|
| UI 变更频率 | 低（稳定的内部工具） | 高（频繁迭代的产品） | N/A |
| 测试执行频率 | 每次 CI（速度优先） | 每次 CI（稳定性优先） | 偶尔/一次性 |
| 团队技能 | 有前端开发能力 | 有基本编程能力 | 无需编程 |
| 预算 | 无额外成本 | 有 VLM Token 预算 | 有 LLM Token 预算 |
| 断言需求 | 结构化（DOM 状态） | 视觉化（"看起来对"） | 自然语言描述 |
| 维护成本 | 高（选择器维护） | 低（自然语言稳定） | 最低（不维护代码） |
| 调试需求 | Playwright Trace | Midscene Report | Agent 对话日志 |

### 5.4.3 混合策略推荐

对于大多数中型项目，推荐的测试金字塔是：

```
        /\
       /  \     少量 Midscene Auto Planning
      /    \    （复杂交互流程的冒烟测试）
     /──────\
    /        \   核心用例用 Midscene Workflow
   /          \  （高变更风险的 UI 交互+视觉断言）
  /────────────\
 /              \ 大量 Playwright 原生用例
/                \（稳定页面的精确验证+API 测试）
──────────────────
```

## 5.5 可落地的 Midscene 测试策略方案

这是本章的核心产出——一份可以直接应用于项目的策略文档。

### 策略方案

#### 1. 引入范围

| 阶段 | 行动 | 时间线 |
|------|------|--------|
| 试点 | 选 2-3 个高维护成本的测试用例用 Midscene 重写 | 第 1-2 周 |
| 扩展 | 新增测试用例默认用 Midscene；脆弱用例逐步迁移 | 第 3-8 周 |
| 稳定 | 建立混合测试套件，确定各方案的分工 | 第 8 周+ |

#### 2. 模型选择

| 用途 | 推荐模型 | 原因 |
|------|---------|------|
| 默认（定位） | Qwen3-VL-Plus | 性价比高，定位准确 |
| 规划 | Doubao Seed 1.6 | 复杂规划能力强 |
| 洞察 | Qwen3-VL-Lite | 简单断言用便宜模型 |

#### 3. 缓存策略

```
开发环境：cache.strategy = "read-write"
  → 开发时快速迭代

CI 环境：cache.strategy = "read-write" + 持久化到 GitHub Cache
  → 首次慢，后续快

Staging 环境：cache.strategy = "write-only"（定期刷新缓存）
  → 确保缓存不会掩盖真实问题
```

#### 4. 成本控制

| 措施 | 预期节省 |
|------|---------|
| 启用缓存 | ~50% Token |
| `screenshotShrinkFactor: 2` | ~30% 输入 Token |
| 非关键路径用 Playwright 原生 | 减少 AI 调用量 |
| `freezePageContext` 批量查询 | 减少截图次数 |
| 多模型策略（便宜模型做简单任务） | ~20% 成本 |

#### 5. CI/CD 配置

```yaml
# 核心原则
timeout: 120s       # 预留充足时间给 VLM
retries: 2          # 应对 AI 偶发不稳定
workers: 2          # 控制并行度
```

#### 6. 监控与告警

- VLM API 错误率 > 5% → 告警
- 单次测试 Token 消耗 > 预算 2x → 告警
- 缓存命中率 < 30% → 检查 UI 是否大幅变更

#### 7. 代码规范

```typescript
// ✅ Good: 描述精确、有上下文
await agent.aiTap('the blue "Submit Order" button at the bottom of the form');

// ❌ Bad: 描述模糊
await agent.aiTap('the button');

// ✅ Good: 需要精确值时先提取再断言
const price = await agent.aiNumber('the total price shown');
expect(price).toBeCloseTo(29.99, 2);

// ❌ Bad: 把精确数值验证交给 AI 断言
await agent.aiAssert('the total price is $29.99');
// AI 可能把 $29.98 也判为 true
```

#### 8. 团队分工

| 角色 | 职责 |
|------|------|
| QA Engineer | 编写 YAML 脚本；维护 Midscene 测试用例 |
| Frontend Dev | 维护 Playwright fixture；处理 Midscene 与原生的混合 |
| DevOps | CI/CD 集成；缓存持久化；成本监控 |
| Tech Lead | 决定哪些用例用 Midscene vs 原生 |

## 思考题

### Q1：如果 VLM 提供商突然宕机或涨价 10 倍，你的 Midscene 测试套件如何应对？需要什么级别的预案？

**答案**：

这是引入 AI 驱动测试的核心风险之一。应对策略分三个层次：

**短期应急（分钟级）**：
- 缓存命中的测试用例不受影响（不调 VLM 就能跑）
- CI 中设置合理的超时和重试，部分用例可能因超时失败，但不会全军覆没

**中期切换（小时级）**：
- Midscene 支持多模型，通过环境变量切换：
```bash
# 从 Qwen 切换到 Doubao
export MIDSCENE_MODEL_BASE_URL=https://ark.cn-beijing.volces.com/api/v3
export MIDSCENE_MODEL_NAME=doubao-seed-1.6-vision
export MIDSCENE_MODEL_FAMILY=doubao-vision
```
- 建议预先测试至少两个 VLM 提供商，确认兼容性

**长期预案**：
- 自建 VLM（开源模型如 UI-TARS、GLM-4.6V 可本地部署）
- 关键路径保留 Playwright 原生备份用例
- 测试策略文档中明确：哪些用例是"AI-only"，哪些有原生备份

**涨价应对**：
- 重新评估成本模型，可能需要减少 AI 驱动的用例数量
- 将更多简单断言降级为 Playwright 原生
- 提高缓存命中率以减少调用

### Q2：Midscene 的测试结果是概率性的（VLM 可能偶尔误判）。在 CI/CD 中，如何区分"真正的 bug"和"AI 的误判"？

**答案**：

这是 AI 驱动测试的核心挑战。实用策略：

**1. 重试策略**：
```typescript
// playwright.config.ts
retries: process.env.CI ? 2 : 0,
```
如果一个测试在重试后通过，大概率是 AI 误判而非真正 bug。

**2. 报告分析**：
- 真正的 bug：连续多次运行都失败，截图显示 UI 确实有问题
- AI 误判：偶发失败，截图显示 UI 正常，VLM 的定位/判断明显错误

**3. 双重验证**：
```typescript
// 关键断言用 Midscene + Playwright 双重验证
await agent.aiAssert('checkout button is visible');
// 原生验证作为兜底
await expect(page.locator('[data-testid="checkout"]')).toBeVisible();
```

**4. 失败分类**：
- 在 CI 报告中标记 Midscene 测试，与原生测试区分
- Midscene 测试的失败不阻塞部署（warning），但记录到监控
- 原生测试的失败才阻塞部署（error）

**5. 趋势监控**：
- 跟踪每个 Midscene 测试用例的 flaky rate
- Flaky rate > 10% → 优化描述或降级为原生

### Q3：回顾整个课程，如果只能给同事说一句话来概括 Midscene 的价值，你会怎么说？

**答案**：

> **Midscene 让你的 E2E 测试像人一样"看"界面而不是"读"代码，所以 UI 怎么改测试都不用动。**

这句话抓住了三个核心点：
1. **"看"界面**——纯视觉路线，不依赖 DOM
2. **"不是读代码"**——与选择器驱动的本质区别
3. **"UI 怎么改测试都不用动"**——最直接的实用价值

当然这是理想情况。更诚实的版本：

> **Midscene 用 AI 视觉理解替代 CSS 选择器，大幅降低 UI 变更导致的测试维护成本，代价是每步操作多几秒延迟和一点 Token 费用。**
